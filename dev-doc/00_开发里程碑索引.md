## 0. 为什么用里程碑制替代顺序开发

原 `00_分功能开发文档索引.md` 推荐开发顺序 01→02→…→10，把 LangGraph 工作流（F10）放最后。但 F10 §0 自身要求"从第 10 个文档开始重整后端主线"。两处自相矛盾，导致团队不知该先做业务功能还是先搭工作流骨架。

里程碑制以 **"最小可演示智能体闭环"** 为第一目标，按"先搭骨架接真实数据 → 再加智能体能力 → 最后前端串联"的路径推进。每个里程碑有明确的验收断言，vibecoder 照着里程碑文档写即可，不需要自行决策开发顺序。

---

## 1. 功能拆分原则（保留）

1. 每个功能单元都有清晰输入、输出、数据边界和验收标准。
2. 每个功能单元可以通过单元测试、接口测试或页面测试独立验证。
3. 规则能完成的诊断不依赖大模型；大模型只用于语义判断和建议生成。
4. 大模型不能直接修改 Excel、SQLite 或 Qdrant 数据。
5. 所有修改动作必须经过结构化校验、人工确认、版本保存和日志记录。
6. 原始 Excel 永远保留，导出文件和新版本单独生成。
7. **LangGraph node 必须薄——只调 service、更新 state、决定下一步。业务逻辑全部在 service 层。**
8. **节点函数不超过 30 行——超出说明把业务逻辑写进了 node。**

## 4. M1：工作流骨架接真实数据（确定性闭环）

### 4.1 里程碑目标

将 graph 的硬编码节点替换为真实 service 调用，跑通"上传→解析→建树→保存版本→结构诊断→报告"的确定性闭环。

### 4.2 智能体特征标注

**本里程碑无智能体特征，为确定性闭环。** 所有节点均为工作流节点（确定性操作），不调用 LLM、不调用 Qdrant、不涉及 tool calling。但为后续智能体节点提供真实数据基础。

### 4.8 验收标准

1. 上传样例 Excel 后，调 `POST /api/workflows/taxonomy/start`，返回 `task_id`
2. 调 `GET /api/workflows/{task_id}` 能看到真实进度（非硬编码 progress 值）
3. 结构诊断检测到 44 个父节点缺失（来自真实数据，非硬编码 44）
4. 生成 `v1.0` 版本记录，`node_count = 21090`
5. 生成 Markdown 报告（模板化，非 LLM 生成），包含真实统计数据
6. `task_record` 表有 `workflow_id`/`thread_id`/`progress` 字段
7. `workflow_event` 表有节点流转记录

### 4.9 禁止行为

- **禁止**在 node 函数中写业务逻辑（树构建算法、诊断规则等），必须调 service
- **禁止**在 M1 阶段调用 LLM 或 Qdrant
- **禁止**硬编码 `structure_issue_count = 44` 等固定值
- **禁止**跳过 `save_initial_version_node` 直接做诊断
- **禁止**在 report_node 中调 LLM（M1 阶段报告是模板化）
- **禁止** node 函数超过 30 行

## 5. M2：向量索引 + 内容诊断智能体（核心智能体能力）

### 5.1 里程碑目标

实现内容诊断的 agent loop，体现 LLM 决策 + tool calling + 迭代推理。这是项目从"工作流"变成"智能体"的转折点。

### 5.2 智能体特征标注

🤖 **LLM 作为决策者 + tool calling + ReAct 循环。** 内容诊断节点不再是单次 service 调用，而是封装了 ReAct 循环的 agent loop。LLM 通过 tool calling 自主决定查询哪些信息（召回相似节点、查父节点路径、查子节点），并迭代推理后输出诊断结论。

### 5.3 涉及节点清单

| 节点名 | 性质 | 调用的 service | 说明 |
|--------|------|---------------|------|
| `index_vector_node` | 工作流·确定性 | `vector_index_service.index_version(version_id)` | 将节点文本写入 Qdrant |
| `diagnosis_planning_node` | 🤖智能体·LLM 规划 | `DiagnosisPlanningAgent.run(structure_stats, tree_overview)` | LLM 规划诊断范围和优先级 |
| `content_diagnosis_node` | 🤖智能体·ReAct loop | `ContentDiagnosisAgent.run(version_id, candidates)` | 召回→LLM判断→补充查询→再判断 |

> `diagnosis_planning_node` 为新增节点，位于 `structure_diagnosis_node` 之后、`content_diagnosis_node` 之前。

## 10. 跨里程碑共享知识

### 10.1 API 规范

- **对外唯一入口**：`/api/workflows/*`（start / status / resume / events）
- **内部辅助 API**：`/api/diagnosis/*`、`/api/suggestions/*`、`/api/taxonomy/*` 等仅供调试和前端查询使用，不作为 workflow 启动入口
- **文件上传 API**：`/api/files/upload` 是 workflow 的前置输入，不属于 workflow API
- 所有 API 响应统一格式：`{code, data, message}`（可选，当前为直接 JSON）

### 10.2 State 规范

- State 定义以 `10_LangGraph智能体工作流开发设计.md` §5.1 的 30+ 字段版为准
- 技术架构文档 §6.3.2 的 10 字段版已废弃，以 F10 版为准
- thread_id 规则：`taxonomy_workflow:{workflow_id}`
- workflow_id 规则：`taxonomy_workflow_{file_id}_{timestamp}`

### 10.3 节点规范

- 所有节点输入 `TaxonomyGraphState`，返回 `state.model_copy(update={...})`
- 节点函数不超过 30 行——超出说明把业务逻辑写进了 node
- 节点不写 SQL、不直接读写 Excel、不直接写 Qdrant、不拼 prompt
- 节点通过 `workflow_event_service.record_step()` 写事件
- 失败节点通过 `node_guard` 包装统一异常处理

### 10.4 智能体节点规范

- 智能体节点（`diagnosis_planning_node`、`content_diagnosis_node`、`generate_suggestion_node`、`generate_report_node`）内部封装 agent loop
- Agent loop 通过 LangChain `create_react_agent` 或手写 while 循环实现
- LLM 通过 `@tool` 装饰的 tool calling 自主决定查询哪些信息
- LLM 不能直接执行 SQL、不能直接写库——通过 `submit_*` tool 提交，由 service 写库
- Agent loop 的 Thought-Action-Observation 链记入日志，供调试和课程演示

### 10.5 数据安全规范

- 大模型只负责分析和生成建议，不直接修改数据库
- 所有修改动作必须结构化（`AdjustmentAction` schema）
- 所有修改动作必须经过程序校验（`validate_action`）
- 中高风险修改必须经过人工确认（interrupt + resume）
- 原始文件必须保留
- 每次修改必须生成新版本
- 所有操作必须写入 `operation_log`

### 10.6 幂等规范

- 每个 service 写操作必须带 `batch_id` 或 `idempotency_key`
- 每个 node 进入前检查 `completed_steps`
- `save_initial_version_node` 对同一 `file_id` 不重复创建 v1.0
- `execute_action_node` 对同一 `review_batch_id` 不重复执行
- `save_new_version_node` 对同一 `action_batch_id` 不重复生成版本
