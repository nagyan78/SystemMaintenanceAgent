# 当前开发路线图

> 状态：CURRENT  
> 更新日期：2026-07-11  
> 本路线替代旧 M1～M5 和 13～16 作为“当前先做什么”的裁决依据。旧编号仅用于追溯。

## 1. 路线原则

- 优先补齐用户闭环，不继续扩张平台概念；
- 优先结果完整性和可追溯性，再做并发、记忆和模型路由；
- 每一阶段必须产生用户可见成果；
- 当前课程项目不引入分布式队列、完整模型网关或大规模多 Agent 平台。

## 2. R1：可信诊断与完整结果

目标：用户上传体系后，能看到覆盖范围明确、可定位、可解释的完整问题清单。

范围：

- 补齐 taxonomy 和 diagnosis 查询 API；
- 建立“全量轻筛查 → 候选召回 → Agent 深诊断”漏斗；
- 记录扫描节点数、候选数、深诊断数、问题数和未诊断原因；
- 让 DiagnosisPlan 的范围和问题类型真正约束执行；
- 问题详情包含节点、路径、规则/语义证据、置信度和来源。

验收：固定样例的已知问题召回率可计算；报告中的每个问题都能跳转到节点和证据。

## 3. R2：安全变更与版本闭环

目标：审核通过的建议能安全执行，并生成可验证的新版本。

范围：

- 统一 workflow run 证据链；
- 审核页展示修改前/后预览、风险和影响节点；
- 补齐必要动作，明确当前不支持的动作；
- 事务执行、幂等键、失败回滚和操作日志；
- 新版本重新索引、重新诊断和质量对比；
- 用户可基于任一版本继续维护。

验收：从 v1.0 生成 v1.1，验证分数和问题变化，再以 v1.1 生成下一次维护任务。

## 4. R3：报告、导出与演示闭环

目标：报告不只是统计摘要，而是一次维护运行的完整审计结果。

报告必须包含：

- 输入文件、输入版本、输出版本和 workflow run；
- 扫描覆盖率与诊断策略；
- 结构问题、内容问题及典型证据；
- 建议、审核决策和拒绝原因；
- 已执行动作、失败动作及影响范围；
- 修改前后统计、质量评分和验证结论；
- 新版本 Excel、diff 和后续维护入口。

验收：一条端到端脚本和一套固定演示数据可重复生成相同结构的报告与导出物。

## 5. R4：可靠性与有限智能增强

只在 R1～R3 完成后实施：

- 统一失败路由、取消和终态不变量；
- run/work item/event 持久化；
- SSE 事件 ID、断点续传和前端去重；
- 有界并发、重试、预算和模型 fallback；
- 基于历史审核的有限业务记忆；
- Golden set 和独立质量评价。

## 6. 暂缓项

- 泛化聊天机器人；
- 六个常驻 Agent 同时协作；
- 分布式 worker/队列；
- 完整模型网关；
- 无明确验收指标的长期记忆；
- 向用户展示原始 chain-of-thought；
- 为展示“智能体感”而增加无业务价值的节点。

## 7. 当前下一步

先完成 R1。推荐开发顺序：

1. 定义 workflow run 级结果 schema 和覆盖率指标；
2. 补 taxonomy/diagnosis API；
3. 重构候选漏斗并补覆盖测试；
4. 修复 planning 与执行不一致；
5. 将问题证据接入前端和报告；
6. 用固定样例执行端到端验收。
