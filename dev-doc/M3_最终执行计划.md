# M3 最终执行计划：建议生成智能体 + 人工审核闭环

> 适用范围：M3 里程碑代码实现与验收。
> 阶段边界：M3 只生成、审核并校验维护建议；不执行动作、不生成新版本、不导出 Excel。动作执行与版本管理属于 M4。

---

## 1. M3 要完成什么

M3 的目标是把 M1/M2 产出的 `diagnosis_issue` 转换成可审核、可校验、可进入 M4 执行阶段的结构化维护建议。

M3 主链路：

```text
diagnosis_issue
  -> generate_suggestion_node
  -> SuggestionAgent ReAct loop
  -> adjustment_suggestion
  -> wait_human_review_node interrupt
  -> GET /api/reviews/{review_batch_id}
  -> POST /api/workflows/{task_id}/resume
  -> validate_action_node
  -> generate_report_node
```

M3 明确不做：

```text
execute_action_node
save_new_version_node
Excel export
rollback
version diff
```

这些从 M4 开始接入。

---

## 2. 架构亮点

1. SuggestionAgent 不是单次 LLM 调用，而是 ReAct/tool calling loop。
2. LLM 只能生成建议，不能直接修改分类树、SQLite、Excel 或 Qdrant。
3. 所有建议必须先经过 `validate_action` tool 自校验，再通过 `submit_suggestion` 结构化入库。
4. 高风险动作必须进入人工审核，workflow 通过 LangGraph `interrupt/resume` 暂停和恢复。
5. `validate_action_node` 只校验 approved 建议是否可被 M4 执行，不在 M3 执行动作。

---

## 3. Graph 拓扑

M3 应将当前 M2 的：

```text
content_diagnosis_node -> generate_report_node
```

改为：

```text
content_diagnosis_node -> generate_suggestion_node -> wait_human_review_node
```

审核后路由：

```text
wait_human_review_node
  -> reject/no approved actions -> generate_report_node
  -> has approved actions -> validate_action_node
```

校验后路由：

```text
validate_action_node
  -> validation failed -> wait_human_review_node 或 failed
  -> validation passed -> generate_report_node
```

M3 禁止接入：

```text
validate_action_node -> execute_action_node -> save_new_version_node
```

## 4. 实现顺序

### 阶段 1：Schema + DB + Repository

1. 新建 `backend/app/schemas/suggestion.py`
   - `AdjustmentAction`
   - `ReviewRequest`
   - `ResumeRequest`
   - `ReviewDecisionResult`
   - `ValidationResult`

2. 修改 `backend/app/db.py`
   - 确保 `adjustment_suggestion` 表存在 M3 所需字段。
   - 使用 `_ensure_columns` 增加 `review_batch_id TEXT`。

3. 新建 `backend/app/repositories/suggestion_repo.py`
   - `create_suggestion`
   - `list_suggestions`
   - `list_by_review_batch`
   - `update_suggestion_status`
   - `update_suggestion_payload`

4. 新建 `backend/app/repositories/operation_log_repo.py`
   - `create_log`
   - `list_logs`

### 阶段 2：Tools + Prompts

5. 新建或补全 `backend/app/tools/validation_tools.py`
   - `validate_action(action_json: str) -> dict`
   - `submit_suggestion(suggestion: dict) -> str`

6. 修改 `backend/app/agents/prompts.py`
   - 增加 `SUGGESTION_SYSTEM_PROMPT`
   - 增加 `SUGGESTION_FEW_SHOT`
   - 明确要求 LLM 先查上下文、再生成建议、再调用 `validate_action`、最后调用 `submit_suggestion`

### 阶段 3：SuggestionAgent

7. 新建 `backend/app/services/suggestion_service.py`
   - `SuggestionAgent.run(version_id, issues)`
   - 规则型问题直接生成建议：`missing_parent`、`deep_level`、`wide_node`、`duplicate_name`
   - LLM 型问题走 ReAct loop：`synonym_pollution`、`semantic_duplicate`、`bad_parent_child_relation`、`naming_irregular`
   - 工具复用 M2 的 `get_node_detail`、`get_node_path`、`get_children`、`search_similar_nodes`
   - 新增工具使用 `validate_action`、`submit_suggestion`
   - 保留 trace：Thought、Action、Observation、validation_result、retry_reason

8. SuggestionAgent 异常处理
   - 未配置 DeepSeek key：规则型建议仍可生成，LLM 型建议标记 skipped。
   - 未知 tool：记录 tool error，不让 workflow 崩溃。
   - 校验失败：最多重试 3 次。
   - 超过重试：记录 `SUGGESTION_GENERATION_FAILED`，继续处理下一条 issue。

### 阶段 4：Review + Action Validation

9. 新建 `backend/app/services/review_service.py`
   - `create_review_request(version_id, review_batch_id)`
   - `apply_review_decision(review_batch_id, decision)`
   - 更新建议状态：`approved`、`rejected`、`edited`
   - 写入 `operation_log`

10. 新建 `backend/app/services/action_service.py`
    - `validate_approved_actions(review_batch_id)`
    - 校验 approved 建议，不执行动作。
    - 校验规则包括：
      - `action_type` 必须合法。
      - `issue_id` 必须存在。
      - `target_node_id` 必须存在，`add_node` 缺失父节点场景除外。
      - `risk_level` 必须是 `low`/`medium`/`high`。
      - `confidence` 必须在 0 到 1。
      - `clean_synonym` 的待删除同义词必须存在于原 `syn_list`。
      - `move_node` 不能移动到自身子树下。
      - `merge_node` 必须包含源节点和目标节点。

### 阶段 5：Node + Graph

11. 修改 `backend/app/agents/nodes.py`
    - `generate_suggestion_node` 调 `SuggestionAgent.run()`，设置 `suggestion_count`、`review_batch_id`。
    - `wait_human_review_node` 调 `review_service.create_review_request()`，然后 `interrupt()`。
    - resume 后调用 `review_service.apply_review_decision()`。
    - `validate_action_node` 调 `action_service.validate_approved_actions()`。
    - node 函数保持薄，不拼 prompt，不写复杂业务逻辑。

12. 修改 `backend/app/agents/graph.py`
    - 改 `content_diagnosis_node -> generate_report_node` 为 `content_diagnosis_node -> generate_suggestion_node`。
    - 增加 `generate_suggestion_node -> wait_human_review_node`。
    - `wait_human_review_node` 条件路由到 `generate_report_node` 或 `validate_action_node`。
    - `validate_action_node` 条件路由到 `wait_human_review_node` 或 `generate_report_node`。
    - M3 不接 `execute_action_node`、`save_new_version_node`。

### 阶段 6：API + Checkpointer

13. 新建 `backend/app/api/reviews.py`
    - `GET /api/reviews/{review_batch_id}` 返回待审核建议列表。

14. 修改 `backend/app/api/workflows.py`
    - 新增 `POST /api/workflows/{task_id}/resume`。
    - resume 时使用 `Command(resume=decision)`。
    - start 和 resume 必须使用同一个 checkpointer 和同一个 `thread_id`。

15. 修改 `backend/app/main.py`

    - 初始化共享 `InMemorySaver` 到 `app.state.checkpointer`。

## 5. 验收标准

1. `generate_suggestion_node` 能读取诊断问题并生成 `adjustment_suggestion`。
2. LLM 型建议日志可见 Thought-Action-Observation 链。
3. LLM 在建议生成过程中调用 `get_node_detail` / `get_node_path` / `search_similar_nodes` / `validate_action` / `submit_suggestion` 中的相关工具。
4. 自校验失败时，Agent 根据失败原因重试；超过上限后记录失败并继续处理下一条 issue。
5. `wait_human_review_node` 能通过 LangGraph `interrupt()` 暂停 workflow。
6. `GET /api/reviews/{review_batch_id}` 能返回 pending 建议列表。
7. `POST /api/workflows/{task_id}/resume` 能恢复 workflow，并将审核决策写入 suggestion 状态和 `operation_log`。
8. `validate_action_node` 能拒绝非法动作，如 move 到自身子树、删除不存在的同义词、非法 action_type。
9. M3 workflow 完成后进入 `generate_report_node`，不进入 `execute_action_node` 或 `save_new_version_node`。
10. M3 不生成新版本，`new_version_id` 为空或未设置。
11. 报告中明确标记：已完成 M3 建议审核校验，待 M4 执行动作并保存新版本。

---

## 6. 禁止行为

- 禁止在 M3 中执行分类树修改。
- 禁止在 M3 中生成新版本。
- 禁止在 M3 graph 主路径中接入 `execute_action_node` / `save_new_version_node`。
- 禁止 LLM 绕过 `validate_action` 直接保存建议。
- 禁止跳过人工审核直接把建议标记为可执行。
- 禁止在 node 函数中拼 prompt 或写复杂校验逻辑。
- 禁止把 M3 的 `/api/suggestions/*` 单条 CRUD 接口作为主验收路径；主验收路径是 interrupt/resume。
