# 产品标准体系维护智能体 — 一页纸价值主张

> 状态：PRODUCT MATERIAL  
> 本文中的目标能力和量化收益不等于当前实现；实现事实以 `../CURRENT_IMPLEMENTATION.md` 为准。

> 用途：课程答辩 / 产品汇报。定位为「从作业到作品」的核心叙事骨架，带 ⚠️ 处为必须补全的诚实缺口。

---

## 一句话定位

给产品分类体系当 **「体检医生」**：在几万行的类目树里，自动找出人眼看不见的脏数据，并给出**可解释、可审核、可回滚**的修复方案。

---

## 我们要解决的真问题

- **对象**：电商/零售类目运营、企业主数据管理（MDM/PIM）、国标/行标产品分类维护团队，常年面对庞大且不断漂移的分类树。
- **核心痛点不是「没有工具」，而是「规则查得出精确重复，查不出语义近似」**：
  - 「智能手机 / 手机 / 智能电话」同义不同名 —— 纯规则脚本永远发现不了；
  - 人工巡检 2 万行类目需要数人天，且靠老员工经验兜底（人员流失即失能）；
  - 周期性人工抽查覆盖低、滞后，脏数据在线上持续产生业务成本。

---

## 我们的方案

```
上传分类体系 Excel
   → 结构诊断（规则：查孤儿节点 / 环路 / 层级错）
   → 内容诊断（LLM + 向量召回：查语义重复 / 归属歧义）
   → 生成带理由的建议
   → 人工审核（人类在环）
   → 执行并保存新版本（原文件永不改）
   → 输出报告
```

---

## 核心差异化（为什么不是又一套规则脚本）

| 差异点 | 说明 | 竞品/传统做法能否做 |
|--------|------|-------------------|
| **语义去重** | Qdrant 向量召回 + LLM 判断「是否应合并」 | 纯规则 ❌ |
| **人类在环（HITL）** | 高风险修改强制人审，新版本独立快照可回滚 | 多数脚本直接改库 ❌ |
| **可解释** | 每条建议带理由，非黑盒结论 | 规则输出常无理由 ⚠️ |

## 价值量化（目标框架，⚠️ 需真实数据补全）

| 维度 | 传统方式（估算基线） | 本系统（目标） |
|------|---------------------|----------------|
| 巡检 2 万行类目 | ~X 人天 | ~10 分钟出候选 |
| 语义重复发现率 | 近 0（靠人偶然发现） | 向量召回覆盖 |
| 误改风险 | 人审疲劳漏看 | 强制 HITL + 可回滚 |

---

## 技术亮点（给评审看）

- **薄节点架构**：LangGraph 作确定性编排骨架，智能体行为收敛在少数节点（诊断规划 / 内容诊断 / 建议生成 / 报告），符合「规则优先、LLM 只做语义判断」原则。
- **可恢复**：SQLite checkpointer 支持 `interrupt/resume`，服务重启后同 `thread_id` 可恢复。
- **可见性**：SSE 实时事件流，前端进度与日志实时可见。
- **安全边界**：LLM 不直接改库 / 不改原 Excel，动作校验 → 人审 → 程序执行 → 存新版本。
