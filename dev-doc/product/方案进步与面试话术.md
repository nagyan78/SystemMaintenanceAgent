# 方案进步说明与面试话术

> 状态：PRODUCT MATERIAL  
> 使用时必须区分“已实现能力”和“目标设计”。六 Agent、work item 账本、Golden set、完整持续维护闭环等仍属于目标架构，不得表述为已全部落地。

> 用途：面试/答辩讲解

## 一、一句话总结演进主线

第一版是一个“上传→诊断→建议→执行→结束”的单次线性流水线。当前已经接入真实 service、语义诊断、建议、人工审核、版本和报告主路径；目标是继续演进为版本驱动、可观测、可恢复、可评价的持续维护闭环，但四阶段目标尚未全部实现。

## 二、第一版长什么样 + 踩了哪些坑

### 2.1 第一版架构（2026-07-04 v1.0）

```
上传 Excel → parse_excel → build_tree → save_v1.0
→ index_vector → structure_diagnosis（纯规则）
→ content_diagnosis → generate_suggestion
→ wait_review（interrupt）→ validate → execute → save_v1.1
→ generate_report → 结束
```

12 个节点固定 DAG，一次性跑完，生成 v1.1 就结束了。

### 2.2 踩的九个坑（按严重程度排序）

| # | 坑 | 具体表现 | 严重程度 |
|---|---|---|---|
| 1 | **像工作流不像智能体** | 12 步线性流水线，唯一的"智能体特征"是 interrupt（流程控制，不是智能体能力）；LLM 完全缺席，零 tool calling，路由全 `if state.xxx` 硬编码 | 🔴 致命 |
| 2 | **关键节点被压成单次调用** | 内容诊断本质是"召回→判断→补充查询→再判断"的迭代循环，被设计成单次 service 调用；建议生成同理 | 🔴 致命 |
| 3 | **无验证闭环** | 执行完动作直接生成报告，不知道修改是否真的改善质量、是否引入新问题 | 🔴 高 |
| 4 | **无规划，全量扫描** | 21090 个节点无差别全量送 LLM，无"先看整体再决定重点" | 🔴 高 |
| 5 | **split 题目要求但不执行** | §10.1 明确"split_subtree 只生成方案，MVP 阶段不自动拆分"——而题目要求过宽子树必须能拆 | 🔴 高 |
| 6 | **不能持续维护** | v1.0→v1.1 就结束，不能基于 v1.1 继续维护到 v1.2 | 🟡 中 |
| 7 | **全硬编码假数据** | nodes.py 返回 `suggestion_count==3`、`structure_issue_count==44`，测试验证的全是写死值 | 🟡 中 |
| 8 | **MemorySaver 重启丢失** | 内存 checkpointer，服务重启状态全丢 | 🟡 中 |
| 9 | **工程细节坑** | DeepSeek 模型名写成 `deepseek-v4-flash`（不存在）；DiagnosisPlan 无上限导致 ReAct 循环疯狂调 API；ChatOpenAI 没设超时 | 🟢 低 |

### 2.3 根因诊断（面试可讲的核心洞察）

> 智能体（Agent）的核心特征不是拓扑结构是否固定，而是 **LLM 是否作为决策者参与流程控制**。LangGraph 固定 DAG 可以是智能体的骨架——前提是关键节点内部封装了 LLM 驱动的 agent loop。第一版的问题是：**骨架有了，但骨架内部的 agent loop 完全不存在。**

第一版的 `route_after_review`：
```python
def route_after_review(state):
    if state.review_decision == "reject":
        return "generate_report_node"
    if state.approved_action_count == 0:
        return "generate_report_node"
    return "validate_action_node"
```
纯规则路由，LLM 完全不参与决策——这就是"像工作流"的根因。

## 三、新方案做了哪些改进

### 3.1 六条改进主线

| # | 第一版 | 新方案 | 驱动原因 |
|---|---|---|---|
| 1 | 单次线性流水线（v1.0→v1.1→结束） | **版本驱动持续维护闭环**（import/maintain/verify 三模式 + version lineage，可 v1.1→v1.2→v1.3） | 真实维护是持续迭代的，不是一次性 |
| 2 | 节点黑盒（跑完返回 count） | **可观测 Agent 执行单元**（work item 账本 + claim/lease + 断点续跑 + 并发隔离） | 第一版崩了全丢，且无法观测 LLM 在做什么 |
| 3 | 全量扫描 21090 节点 | **两阶段智能筛选**（2万→确定性筛查→300候选→50深诊断） | token 预算和延迟，不可能全量跑 LLM |
| 4 | 单维度总分（§18 一个公式） | **六维版本化质量评价 + 验证**（structural/hierarchy/semantic/redundancy/naming/coverage，带 score_version，修改前后对比 resolved/unresolved/introduced） | 第一版不知道修改是否真改善、是否有回归 |
| 5 | split 不执行 + 无 deprecate | **完整动作语义 + 先模拟后执行**（split/merge/deprecate/delete_leaf 全可执行，SnapshotActionApplier 纯内存快照模拟 + preview hash 校验） | 题目要求过宽拆分和冗余合并必须能做 |
| 6 | LLM 缺席 | **6 专职 Agent + ReAct loop + tool calling + 滚动规划 + 审核记忆 + Golden set** | 让系统真正具备智能体能力 |

### 3.2 六个 Agent 的职责分工

| Agent | 职责 | 智能体特征 |
|---|---|---|
| Maintenance Supervisor | 读基线+结构统计+历史，规划诊断范围/预算/停止 | 强（LLM 规划，但预算/停止由程序兜底） |
| Structure Remediation | 生成 split/merge 结构治理方案 | 强（LLM + cluster_children 工具 + 提交前 simulation） |
| Content Diagnosis Worker | 单节点语义判断（召回→判断→补充查询→再判断） | 强（ReAct loop + tool calling） |
| Suggestion | 动作生成 + 自校验（validate 失败最多修复 2 次） | 强（ReAct loop + validate_action 工具） |
| Quality Evaluation | 解释/归因/优先级（**评分由确定性服务算**） | 弱（LLM 只做语言层） |
| Verification/Critic | 验证 + 路由（**路由受确定性策略约束**） | 中（决策被约束） |

### 3.3 确定性 vs 智能体的边界（关键设计原则）

> Agent 只能查询、规划、模拟、解释和提交候选动作；**不能直接修改 category_node、正式版本、Qdrant 或原始 Excel。**

- 规则能解决的不交给大模型（结构诊断是纯规则）
- 评分由确定性 MetricsService 计算，LLM 只解释
- 路由受确定性策略约束（高风险回归→人工介入；达标→结束；有残留→询问继续）
- 所有副作用收口到事务化的 ActionExecutor，LLM 不碰库

## 四、为什么要这么设计（每个改进的驱动原因）

### 4.1 为什么从"单次"改"持续维护"？
第一版执行完 v1.1 就结束。但真实场景里，维护一轮后会发现新问题，需要基于 v1.1 继续维护到 v1.2。而且 maintain 模式**禁止固定回 v1.0**——否则永远在旧版本上修，新版本的修复会丢。

### 4.2 为什么做"两阶段筛选"而不是全量？
21090 个节点逐个跑 LLM，按每节点 3-5 次调用、每次 2-5 秒，总耗时数小时、token 成本不可接受。两阶段：先用确定性规则筛查到 300 候选，再 LLM 深诊断 50 个——这是唯一能在课程项目预算内跑通"检测每一个节点"的方案。

### 4.3 为什么"先模拟后执行"？
审核到执行之间，版本数据可能变化（用户可能又改了东西）。所以执行前**强制重新模拟**，preview hash 不一致就拒绝执行。这防止了"审核的是 A，执行的是 B"的数据漂移。

### 4.4 为什么"评分确定性算、LLM 只解释"？
如果让 LLM 算分，同一个版本每次评分可能不同（温度、采样），无法做"修改前后对比"。所以评分用固定公式 `quality-v1`，带版本号可迭代；LLM 只负责把数字翻译成人话、归因和排优先级。

### 4.5 为什么做 Golden set 独立评估？
不能让 Agent 自己评价自己（既当运动员又当裁判）。Golden set 是**人工标注的期望结果**，不把模型预测反写成 golden truth。课程 demo 强制 `unsafe action escape=0`（危险动作漏拦截率必须为 0），这是硬门禁。

### 4.6 为什么审核记忆分 confidence？
用户反馈 `confidence=1.0`（事实），模型推断记忆 `≤0.7`（不绝对可靠）。记忆只影响生成和排序，**不能跳过本轮 validate/human review**——否则 LLM 会用"上次被拒"当借口跳过审核。

## 六、面试怎么讲（叙事框架）

### 6.1 三段式叙事

**第一段：项目是什么（30 秒）**
> 这是一个产品标准分类体系的维护智能体。用户上传一个 2 万节点的 Excel 分类树，系统像"分类体系医生"一样体检——结构诊断用规则、内容诊断用 LLM + 向量召回，给出可解释、可确认、可执行、可回滚的维护建议，人工审核后执行，生成新版本和报告。

**第二段：第一版踩了什么坑（90 秒，重点讲"像工作流不像智能体"）**
> 第一版用 LangGraph 画了一个 12 节点的固定 DAG，从上传到报告一次性跑完。但做完发现一个致命问题：**它像工作流不像智能体**。唯一的"智能体特征"是 interrupt 等待人工审核，但 interrupt 是流程控制，不是智能体能力。LLM 完全缺席，零 tool calling，路由全是 `if state.xxx` 硬编码。
>
> 根因是：**智能体的核心特征不是拓扑结构是否固定，而是 LLM 是否作为决策者参与流程控制。** 我有骨架，但骨架里的 agent loop 完全不存在。内容诊断本该是"召回→判断→补充查询→再判断"的迭代循环，被压成了单次 service 调用。
>
> 另外还有几个工程坑：21090 节点无差别全量扫描、执行完不验证是否真改善、split 题目要求但"MVP 不自动拆分"、MemorySaver 重启丢状态。

**第三段：怎么调优的（120 秒，讲六条改进主线 + 一个核心原则）**
> 先做了一次架构评审，确立了"确定性骨架 + 智能体节点"的分层原则——规则能解决的不交给大模型，LLM 只做语义判断、规划、解释，不碰库。然后分四阶段调优：
>
> 1. **版本闭环**：从单次流水线改成 import/maintain/verify 三模式，可以基于任意版本继续维护，maintain 禁止固定回 v1.0。
> 2. **执行单元化**：把节点黑盒拆成可观测的 work item，每个候选是一个账本记录，有 claim/lease/submit，崩了能断点续跑。
> 3. **两阶段筛选**：2 万节点 → 确定性筛查到 300 候选 → LLM 深诊断 50 个，token 成本降两个数量级。
> 4. **质量评价 + 验证**：六维评分带版本号，修改前后对比，算出 resolved/unresolved/introduced，知道是否真改善、是否有回归。
> 5. **动作补齐 + 先模拟后执行**：split/merge/deprecate/delete 全可执行，执行前在内存快照上重新模拟，preview hash 不一致就拒绝。
> 6. **智能增强**：滚动规划（批次反馈驱动扩圈/缩圈，但预算和停止由程序兜底）、模型路由、审核记忆、Golden set 独立评估。
>
> 核心设计原则一句话：**评分确定性算、路由策略约束、LLM 只解释不碰库、所有副作用事务化执行。**

### 6.2 可能被追问的问题 + 回答要点

**Q：为什么不直接用 LangGraph 的 Send API 做全并发？**
A：阶段一先用节点内串行循环跑通闭环，阶段二才引入 Send/map-reduce。因为 work item 账本和 lease 恢复机制要先建好，否则 Send fan-out 崩了无法恢复。这是"先正确再并发"的工程顺序。

**Q：LLM 幻觉怎么防？**
A：四道防线——①结构化输出 + Pydantic 校验；②validate_action 工具自校验，失败最多修复 2 次；③先模拟后执行，preview hash 校验防数据漂移；④高风险动作强制人工审核，LLM 不能绕过。课程 demo 强制 unsafe action escape=0。

**Q：2 万节点怎么全覆盖？**
A：两阶段。先确定性规则筛查（结构问题、同义词非空、过宽过深），筛到约 300 候选；再 LLM 深诊断 50 个高优先候选。这样"检测了每一个节点"（筛查阶段全覆盖），同时 LLM 只在最需要的地方花 token。

**Q：版本怎么保证不重复创建？**
A：save_new_version 用 action_batch_id 做幂等键。同一个审核批次恢复执行时，已存在则返回原版本，不会重复创建 v1.2。

**Q：质量评分为什么不让 LLM 算？**
A：因为同一个版本每次 LLM 评分可能不同（温度、采样），无法做"修改前后对比"。评分用固定公式 quality-v1，带版本号可迭代；LLM 只负责把数字翻译成人话、归因和排优先级。

**Q：这个项目最让你骄傲的点是什么？**
A：确定性边界划得清楚。很多人做 Agent 项目容易陷入"什么都交给 LLM"的陷阱，结果不可控、不可复现。这个项目把规则诊断、质量评分、路由决策、动作执行全部收口到确定性组件，LLM 只在"需要语义判断"的地方出手——这让系统既体现了智能体能力，又保证了工程可控性。

## 七、一页纸核心卖点（面试最后总结用）

1. **确定性骨架 + 智能体节点的分层架构**——规则能解决的不交给大模型
2. **版本驱动的持续维护闭环**——不是一次性，能 v1.1→v1.2 迭代，有验证闭环
3. **先模拟后执行**——SnapshotActionApplier + preview hash 校验，防数据漂移
4. **两阶段智能筛选**——2万节点全覆盖，LLM 只在 50 个高优先候选上花 token
5. **Golden set 独立评估**——不让 Agent 自评，unsafe action escape=0 硬门禁
6. **可观测可恢复**——work item 账本 + lease 恢复 + SSE 事件流
