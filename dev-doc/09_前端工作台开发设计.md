# 前端工作台开发设计

> 功能编号：F09
> **治理更新（2026-07-17）**：移除审核页、审核路由和审核 API。前端只展示自动工作流、版本和报告；建议的动作、置信度、状态及跳过/失败原因在报告中以只读“自动决策摘要”呈现。文中旧审核交互说明均已废止。
> **里程碑归属：M5（前端工作台 + 端到端演示 + 可选 Electron 桌面壳）**
> 独立测试目标：通过本地工作台完成上传、智能体分析、人工审核、动作执行、新版本查看、diff、导出和报告预览的完整演示流程。
> 相关源需求：PRD 9，技术架构 6.1、17。

---

## 1. 功能目标

提供一个面向体系管理员、业务专家和课程答辩用户的本地智能体工作台。前端只负责展示和交互，所有 Excel 处理、诊断、智能体流程、版本保存和导出均由后端 API 完成。

M5 的价值不是新增智能体能力，而是把 M1-M4 的能力可视化：

1. 用户能上传 Excel 并启动 LangGraph workflow。
2. 用户能看到每个工作流节点的执行状态。
3. 用户能在 `waiting_review` 阶段审核 Agent 生成的维护建议。
4. 用户恢复 workflow 后能看到动作执行、新版本生成、版本 diff、导出和报告。
5. 课程或面试演示中能清晰展示：Agent 规划、Tool Calling、Human-in-the-loop、版本化治理闭环。

---

## 2. 当前前端状态

当前仓库根目录下没有稳定可用的 `frontend/` 目录，M5 应按新建前端工作台处理。

推荐技术栈：

```text
Vue 3 + Vite + TypeScript + Vue Router + 原生 CSS/Tailwind（任选其一）
```

不建议 M5 一开始引入复杂状态管理库。工作台状态可以先用轻量 `workspace.ts` 模块 + `localStorage` 管理。

## 3. M5 总体用户流程

```text
/upload
  上传 Excel
  -> POST /api/files/upload
  -> POST /api/workflows/taxonomy/start
  -> 保存 file_id/task_id/thread_id

/workflow/:taskId
  展示 workflow 节点进度
  -> GET /api/workflows/{task_id} 轮询
  -> 如 status=waiting_review，跳转 /review/:reviewBatchId?task_id=xxx

/review/:reviewBatchId
  展示 Agent 建议
  -> GET /api/reviews/{review_batch_id}
  -> 用户 approve/reject/edit
  -> POST /api/workflows/{task_id}/resume

/versions
  展示 v1.0/v1.1/...
  -> GET /api/versions?file_id=xxx
  -> GET /api/versions/{id}/diff?target_version_id=xxx
  -> GET /api/versions/{id}/export

/report/:versionId
  展示 Markdown 报告或报告文件路径
```

M5 MVP 可以先用轮询，不强制 SSE。如果实现 SSE，则作为增强项。

## 4. 页面结构

| 页面 | 路由 | 优先级 | 核心能力 |
|---|---|---:|---|
| 上传与启动页 | `/upload` | P0 | 上传 Excel、展示字段识别、启动 workflow |

| 版本管理页 | `/versions` | P0 | 版本列表、选择对比、diff、导出、回滚 |
| 报告页 | `/report/:versionId` | P0 | 展示报告路径/Markdown 内容、下载入口 |
| 体系概览页 | `/overview/:versionId` | P1 | 统计卡片、一级类目、质量分 |
| 分类树页 | `/tree/:versionId` | P1 | 树浏览、搜索、节点详情 |
| 诊断问题页 | `/diagnosis/:versionId` | P1 | 结构/内容问题列表与筛选 |
| Agent Trace 页 | `/agent-trace/:taskId` | P2 | 展示 Thought/Action/Observation 摘要 |
| Electron 设置页 | `/settings` | P2 | 本地后端地址、Qdrant/LLM 配置状态展示 |

## 6. 全局工作区状态

`frontend/src/state/workspace.ts` 保存当前工作上下文：

```ts
export type WorkspaceState = {
  apiBaseUrl: string
  fileId: number | null
  fileName: string | null
  taskId: string | null
  workflowId: string | null
  threadId: string | null
  currentVersionId: number | null
  newVersionId: number | null
  versionNo: string | null
  reviewBatchId: string | null
  reportPath: string | null
}
```

规则：

1. 上传成功后写入 `fileId`、`fileName`。
2. 启动 workflow 后写入 `taskId`、`workflowId`、`threadId`。
3. 轮询 workflow 状态时更新 `currentVersionId`、`newVersionId`、`reviewBatchId`、`reportPath`。
4. `status=waiting_review` 时跳转审核页。
5. `status=completed` 时跳转版本页或报告页。
6. 刷新页面后从 `localStorage` 恢复工作区状态。

## 7. API 契约

### 7.1 文件上传

```text
POST /api/files/upload
```

返回字段至少包括：

```ts
type UploadResponse = {
  file_id: number
  task_id?: string
  file_name: string
  row_count: number
  column_count: number
  columns: string[]
  status: string
}
```

### 7.2 启动 workflow

```text
POST /api/workflows/taxonomy/start
```

请求：

```json
{ "file_id": 1 }
```

返回：

```ts
type StartWorkflowResponse = {
  task_id: string
  workflow_id: string
  thread_id: string
  status: string
  current_step: string
  progress: number
}
```

### 7.3 查询 workflow 状态

```text
GET /api/workflows/{task_id}
```

前端期望字段：

```ts
type WorkflowStatus = {
  task_id: string
  status: 'pending' | 'running' | 'waiting_review' | 'completed' | 'failed'
  current_step: string
  progress: number
  file_id: number
  current_version_id?: number
  new_version_id?: number
  version_no?: string
  node_count?: number
  structure_issue_count?: number
  content_issue_count?: number
  suggestion_count?: number
  approved_action_count?: number
  executed_action_count?: number
  review_batch_id?: string
  report_path?: string
  error_message?: string
}
```

### 7.4 获取审核建议

```text
GET /api/reviews/{review_batch_id}
```

返回：

```ts
type ReviewBatch = {
  review_batch_id: string
  suggestion_count: number
  suggestions: Suggestion[]
}
```

### 7.5 恢复 workflow

```text
POST /api/workflows/{task_id}/resume
```

请求：

```ts
type ResumeRequest = {
  decision: 'approve' | 'reject' | 'edit'
  approved_suggestion_ids: number[]
  rejected_suggestion_ids: number[]
  edits?: Array<Record<string, unknown>>
  operator?: string
  reject_reason?: string
}
```

### 7.6 版本管理

```text
GET  /api/versions?file_id=1
GET  /api/versions/{version_id}
GET  /api/versions/{version_id}/diff?target_version_id=2
POST /api/versions/{version_id}/rollback
GET  /api/versions/{version_id}/export
```

### 7.7 诊断和体系查询（P1）

如果后端已有接口则直接对接；如果仍是 placeholder，M5 后端补齐：

```text
GET /api/taxonomy/overview?version_id=1
GET /api/taxonomy/tree?version_id=1&parent_id=&depth=2
GET /api/taxonomy/nodes/{node_id}?version_id=1
GET /api/diagnosis/issues?version_id=1&type=&risk_level=&status=
GET /api/suggestions?version_id=1&review_batch_id=&status=
```

### 7.8 报告预览（P1）

如果后端只返回 `report_path`，前端无法直接读取本地文件。建议 M5 补一个报告预览接口：

```text
GET /api/reports/{version_id}/preview
```

返回：

```ts
type ReportPreview = {
  version_id: number
  report_path: string
  markdown: string
}
```

## 8. 工作流节点展示

`WorkflowView` 使用固定节点配置渲染步骤条：

```ts
const workflowSteps = [
  { key: 'parse_excel', label: '解析 Excel', phase: 'M1' },
  { key: 'build_tree', label: '构建分类树', phase: 'M1' },
  { key: 'save_initial_version', label: '保存 v1.0', phase: 'M1' },
  { key: 'index_vector', label: '向量索引', phase: 'M2' },
  { key: 'structure_diagnosis', label: '结构诊断', phase: 'M1/M2' },
  { key: 'diagnosis_planning', label: '诊断规划 Agent', phase: 'M2' },
  { key: 'content_diagnosis', label: '内容诊断 Agent', phase: 'M2' },
  { key: 'generate_suggestion', label: '建议生成 Agent', phase: 'M3' },
  { key: 'wait_human_review', label: '人工审核', phase: 'M3' },
  { key: 'validate_action', label: '动作校验', phase: 'M3' },
  { key: 'execute_action', label: '执行动作', phase: 'M4' },
  { key: 'save_new_version', label: '保存新版本', phase: 'M4' },
  { key: 'completed', label: '生成报告', phase: 'M4' },
]
```

状态映射：

1. `current_step` 之前：completed。
2. `current_step` 当前项：running。
3. `status=waiting_review`：人工审核项显示 waiting。
4. `status=failed`：当前项显示 failed，并展示错误。
5. `status=completed`：全部可用步骤显示 completed。

## 9. 页面设计细节

### 9.1 UploadView

组件：

- `FileDropzone`
- `FileInfoCard`
- `StartAnalysisButton`

交互：

1. 用户选择 `.xlsx`。
2. 调 `POST /api/files/upload`。
3. 显示文件名、行数、列数、字段列表。
4. 点击“开始智能体分析”。
5. 调 `POST /api/workflows/taxonomy/start`。
6. 跳转 `/workflow/:taskId`。

### 9.2 WorkflowView

组件：

- `StepTimeline`
- `AgentCapabilityPanel`
- `TaskSummaryCard`

展示：

- 当前状态
- 当前节点
- 进度百分比
- file_id / version_id / review_batch_id
- 节点阶段标签 M1/M2/M3/M4

跳转规则：

- `waiting_review`：展示“进入审核”按钮，并可自动跳转 `/review/:reviewBatchId?task_id=xxx`。
- `completed`：展示“查看版本”和“查看报告”按钮。
- `failed`：展示错误信息和重试提示。

### 9.3 ReviewView

组件：

- `SuggestionTable`
- `ActionPreview`
- `ReviewToolbar`

列表字段：

- action_type
- target_node_name
- reason
- suggestion
- risk_level
- confidence
- status

操作：

1. 单条勾选 approve/reject。
2. 编辑建议 payload。
3. 批量 approve 低风险建议。
4. 点击“提交审核并继续执行”。
5. 调 `/api/workflows/{task_id}/resume`。
6. 跳回 workflow 页面继续轮询。

### 9.4 VersionsView

组件：

- `VersionTable`
- `VersionDiff`
- `ExportButton`
- `RollbackConfirmDialog`

能力：

1. 显示版本列表。
2. 默认选择最新两个版本做 diff。
3. 展示 added/deleted/renamed/moved/synonym_changed。
4. 导出指定版本 Excel。
5. 回滚版本并刷新列表。

### 9.5 ReportView

MVP：展示 `report_path` 和下载提示。

增强：调用 `GET /api/reports/{version_id}/preview` 展示 Markdown。

组件：

- `MarkdownViewer`
- `ReportMetaCard`
- `DownloadButton`

### 9.6 Overview / Tree / Diagnosis

P1 页面，用于增强演示细节。

Overview：展示统计卡片。

Tree：展示分类树，点击节点看详情。

Diagnosis：展示结构问题和内容问题列表。

如果后端接口不足，M5 可先把这些页面降级为“从报告/任务摘要进入”的只读页面，不阻塞 P0 演示链路。

## 10. Electron 桌面壳设计

Electron 是 M5+ 增强项，不阻塞 Web 工作台验收。

### 10.1 推荐分阶段

#### 阶段 A：轻量桌面壳

```text
用户手动启动 FastAPI 后端
Electron 加载前端 dist
前端通过 http://127.0.0.1:8000/api 调后端
```

优点：实现快，适合演示。

#### 阶段 B：完整桌面应用

```text
Electron main process
  -> 启动 Python FastAPI 子进程
  -> 检查 127.0.0.1:8000/api/health
  -> 打开 Vue 窗口
  -> 退出时关闭后端子进程
```

Qdrant 处理策略：

1. MVP：提示用户先启动 Docker Qdrant。
2. 增强：Electron 检查 Qdrant 端口，不自动管理 Docker。
3. 完整版：提供“一键启动 Qdrant”按钮，但需要用户本机 Docker 可用。

### 10.2 Electron 文件结构

```text
electron/
├── main.ts          # BrowserWindow + backend process lifecycle
├── preload.ts       # 安全暴露少量桌面能力
└── backend.ts       # 启动/停止 FastAPI 子进程
```

### 10.3 桌面配置

前端 API base URL 从以下来源读取：

1. `localStorage.apiBaseUrl`
2. Electron preload 注入
3. 默认 `http://127.0.0.1:8000/api`

## 12. 验收标准

P0 验收：

1. 前端可以上传 Excel 并启动 workflow。
2. 工作流页面能轮询展示状态和当前节点。
3. `waiting_review` 时能展示建议列表。
4. 用户能提交审核并 resume workflow。
5. workflow 完成后能看到新版本。
6. 版本页能查看 diff 并导出 Excel。
7. 报告页能展示报告路径或 Markdown 预览。
8. 所有数据来自后端 API，不硬编码诊断结果。

P1 验收：

1. 体系概览页展示统计卡片。
2. 分类树页支持浏览和节点详情。
3. 诊断页支持问题筛选。
4. 报告支持 Markdown 预览。

P2 验收：

1. Electron 轻量桌面壳可打开前端。
2. Electron 能配置后端 API 地址。
3. Electron 能检测后端健康状态。

## 13. 禁止行为

- 禁止前端直接读写 SQLite。
- 禁止前端直接调用 LLM、Qdrant 或修改 Excel。
- 禁止前端硬编码诊断结果、建议结果、版本 diff。
- 禁止绕过 workflow 直接执行建议。
- 禁止把 Electron 打包作为 M5 P0 前置条件。
- 禁止为了页面展示伪造 `v1.1`、diff 或报告。

## 15. 面试/演示叙事

M5 页面应能支撑这句话：

> 这个项目不是聊天机器人，而是一个本地 Agentic Workflow 工作台。用户上传分类体系 Excel 后，系统通过 LangGraph 编排确定性处理、RAG 内容诊断、建议生成 Agent、人工审核 interrupt/resume、动作执行和版本保存，并在前端完整展示每一步状态和结果。
