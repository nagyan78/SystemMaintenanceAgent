# 智能建议生成开发设计

> 功能编号：F05  
> 独立测试目标：根据结构诊断和内容诊断问题，生成可校验、可审核、可执行的结构化维护建议。  
> 相关源需求：PRD 8.6，技术架构 6.4、10、11.4、12.3。

---

## 1. 功能目标

将 `diagnosis_issue` 转换为 `adjustment_suggestion`。每条建议必须包含问题来源、动作类型、目标节点、原因、建议说明、风险等级、置信度和是否需要人工确认。建议只描述动作，不直接修改节点、数据库或 Excel。

---

## 2. 功能边界

### 2.1 输入

1. `version_id`
2. `diagnosis_issue` 待处理问题
3. 节点详情与上下文路径
4. 相似节点结果，可选
5. LLM 配置，可选

### 2.2 输出

1. `adjustment_suggestion` 记录。
2. 可供前端展示的建议列表。

### 2.3 不包含

1. 不执行建议。
2. 不更新节点版本。
3. 不导出 Excel。
4. 不绕过人工审核。

## 4. 数据模型

```sql
CREATE TABLE adjustment_suggestion (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    issue_id INTEGER NOT NULL,
    version_id INTEGER NOT NULL,
    action_type TEXT NOT NULL,
    target_node_id INTEGER,
    target_node_name TEXT,
    old_parent_id INTEGER,
    new_parent_id INTEGER,
    old_name TEXT,
    new_name TEXT,
    action_payload TEXT,
    reason TEXT,
    suggestion TEXT,
    risk_level TEXT,
    confidence REAL,
    need_confirm INTEGER DEFAULT 1,
    status TEXT DEFAULT 'pending',
    created_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (issue_id) REFERENCES diagnosis_issue(id),
    FOREIGN KEY (version_id) REFERENCES taxonomy_version(id)
);
```

---

## 5. 动作类型

| 动作 | 来源问题 | 说明 | 默认人工确认 |
|---|---|---|---|
| `add_node` | `missing_parent` | 新增缺失父节点或中间分类 | 是 |
| `move_node` | `bad_parent_child_relation` | 将节点迁移到更合理父节点下 | 是 |
| `rename_node` | `naming_irregular` | 修改节点名称 | 是 |
| `merge_node` | `semantic_duplicate`、`duplicate_name` | 合并重复节点 | 是 |
| `clean_synonym` | `synonym_pollution` | 清理错误同义词 | 是 |
| `split_subtree` | `wide_node` | 为过宽节点提出拆分方案 | 是 |
| `mark_as_valid` | 重复但合理复用 | 标记合理，不修改节点 | 否 |

---

## 6. 生成策略

### 6.1 规则型建议

适合确定性问题：

| 问题类型 | 建议动作 | 生成规则 |
|---|---|---|
| `missing_parent` | `add_node` | 从 `category_group_name` 中提取缺失父节点名称，建议补齐 |
| `deep_level` | `mark_as_valid` 或 `move_node` | 默认标记需人工判断，不自动移动 |
| `wide_node` | `split_subtree` | 只生成拆分方案，不自动拆分 |
| `duplicate_name` | `mark_as_valid` | 默认先标记待判断，避免误合并 |

### 6.2 LLM 型建议

适合语义问题：

1. 同义词污染：生成 `clean_synonym`。
2. 父子关系异常：生成 `move_node`。
3. 语义重复：生成 `merge_node` 或 `mark_as_valid`。
4. 命名不规范：生成 `rename_node`。

## 8. API 设计

### 8.1 生成建议

```text
POST /api/suggestions/generate
```

请求：

```json
{
  "version_id": 1,
  "issue_ids": [101, 102, 103]
}
```

响应：

```json
{
  "version_id": 1,
  "generated_count": 3,
  "suggestions": [
    {
      "id": 1,
      "issue_id": 101,
      "action_type": "clean_synonym",
      "target_node_id": 441,
      "target_node_name": "苹果",
      "risk_level": "medium",
      "confidence": 0.92,
      "need_confirm": true,
      "status": "pending"
    }
  ]
}
```

### 8.2 获取建议列表

```text
GET /api/suggestions?version_id=1&status=pending&action_type=clean_synonym
```

### 8.3 获取建议详情

```text
GET /api/suggestions/{suggestion_id}
```

## 9. 校验规则

生成建议后立即做结构校验：

1. `action_type` 必须在允许枚举中。
2. `issue_id` 必须存在。
3. `target_node_id` 必须存在，`add_node` 的缺失父节点场景除外。
4. `risk_level` 必须是 `low`、`medium`、`high`。
5. `confidence` 必须在 0 到 1 之间。
6. `need_confirm=true` 的建议不能自动执行。
7. `clean_synonym` 的待删除同义词必须存在于原 `syn_list`。
8. `merge_node` 必须包含源节点和目标节点。

## 10. 测试设计

### 10.1 单元测试

| 测试项 | 输入 | 期望 |
|---|---|---|
| 同义词污染建议 | `synonym_pollution` 问题 | 生成 `clean_synonym` |
| 过宽节点建议 | `wide_node` 问题 | 生成 `split_subtree` |
| 重复名称建议 | `duplicate_name` 问题 | 生成 `mark_as_valid` 或待判断建议 |
| 非法动作类型 | LLM 返回 `delete_node` | 校验失败 |
| 置信度越界 | `confidence=1.2` | 校验失败 |

### 10.2 接口测试

1. 准备 3 条诊断问题。
2. 调用 `POST /api/suggestions/generate`。
3. 断言 `adjustment_suggestion` 新增 3 条记录。
4. 断言所有高风险动作 `need_confirm=true`。
5. 断言返回 JSON 可被前端直接展示。

---

## 11. 验收标准

1. 每个建议必须包含问题、原因、动作、风险等级和置信度。
2. 建议必须是结构化 JSON，方便程序校验和执行。
3. 高风险动作不能自动执行。
4. LLM 返回非法动作时系统拒绝保存并记录错误。
5. 建议生成不修改任何节点数据。
