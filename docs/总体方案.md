# 标准产品体系维护智能体解决方案

## 1. 方案目标

标准产品体系通常以 Excel 中的多层分类树形式保存。随着体系长期扩充，容易出现层级过深、节点过宽、结构不均衡、重复挂载、父子关系错误、节点冗余、同义词缺失或污染等问题。人工逐条检查成本高，且难以持续记录每次维护的依据和结果。

本方案设计一套标准产品体系维护智能体，完成以下四类核心任务：

1. **结构维护**：检测层级过深、节点过宽、分支不均衡、父节点缺失等结构问题，并提出合并、拆分、补充中间层级等调整方案。
2. **内容优化**：识别冗余节点、重复挂载节点、不合理父子关系、命名不规范和同义词问题，生成可执行的移动、合并、重命名和同义词清理建议。
3. **体系评价**：从结构合理性、内容准确性、同义词覆盖率等维度建立量化指标，比较维护前后的质量变化。
4. **版本管理**：记录每次诊断、审核和调整过程，生成不可变的新版本，支持历史版本对比、回滚和年度/月度差异分析。

系统采用“**规则检测 + 模型判断 + 人工审核 + 程序执行 + 版本验证**”的方式。规则负责全量、低成本筛查；大语言模型负责语义判断和方案生成；程序负责动作校验和数据修改。大语言模型不直接修改原始 Excel、数据库或历史版本。

---

## 2. 总体工作流程

```text
上传或选择标准产品体系 Excel
        ↓
字段校验与分类树解析
        ↓
创建不可变初始版本 v1.0
        ↓
结构类规则全量检测
        ↓
内容类规则筛查、向量召回或抽样分析
        ↓
调用模型进行语义复核
        ↓
生成问题说明、判断依据和调整建议
        ↓
人工审核：接受 / 拒绝 / 编辑
        ↓
程序模拟并校验调整动作
        ↓
执行审核通过的动作
        ↓
生成新版本 v1.1
        ↓
重新诊断并计算维护前后质量变化
        ↓
导出新 Excel、版本差异和诊断报告
```

系统最终输出的不只是“发现多少问题”，还应包括：

- 问题节点名称和节点 ID；
- 节点完整路径；
- 问题类型和风险等级；
- 规则统计或语义判断依据；
- 模型置信度；
- 建议执行的动作；
- 修改前后预览；
- 新旧版本差异；
- 体系质量评分变化。

## 3. 模型选择与调用方式

### 3.1 模型选择

本项目采用“**本地模型测试 + 云端模型最终部署**”的方式。

在开发和调试阶段，使用本地部署的 `qwen3:8b` 模型完成智能体流程验证，降低调用成本，方便反复调整 Prompt 和算法逻辑。

在最终使用阶段，接入 DeepSeek API 作为主要推理模型，提高复杂语义分析、问题判断和维护建议生成的准确性。

| 使用阶段     | 模型                          | 作用                                                  |
| ------------ | ----------------------------- | ----------------------------------------------------- |
| 开发测试阶段 | `qwen3:8b`（Ollama 本地部署） | 验证模型调用流程、Prompt 效果、节点语义分析和建议生成 |
| 最终运行阶段 | DeepSeek API                  | 用于复杂语义判断、内容诊断和维护方案生成              |
| 向量检索     | Embedding 模型                | 生成节点向量，用于相似节点召回和语义匹配              |

模型配置通过 `.env` 管理：

```
# 开发测试
LLM_PROVIDER=ollama
LLM_MODEL=qwen3:8b

# 最终部署
# LLM_PROVIDER=deepseek
# LLM_MODEL=deepseek-chat

EMBEDDING_MODEL=text-embedding-v3

MODEL_TIMEOUT=60
MODEL_MAX_RETRIES=2
```

------

### 3.2 模型调用原则

1. **规则可以确定的问题不调用模型**，例如父节点缺失、名称为空、完全重复节点等问题由程序规则直接判断。
2. **模型只负责语义分析和建议生成**，例如父子关系是否合理、节点是否重复、分类是否需要调整等。
3. **模型输入只包含必要上下文**，不会直接传入完整 Excel 文件。
4. **模型输出采用结构化格式**，方便系统保存、展示和后续审核。
5. **模型生成的修改建议不能直接执行**，需要经过程序校验和人工审核。
6. **开发阶段使用本地模型，最终系统切换为 DeepSeek API**，两种模型保持统一调用接口，方便切换。

## 4. 结构维护方案

结构维护用于检查分类树整体是否合理，重点解决层级过深、节点过宽、分支不均衡、父节点缺失和结构冗余等问题。

### 4.1 树的高度

#### 相对阈值判断

超过平均高度加两倍标准差的路径，视为异常深路径：

$$
子树高度 > mean(H) + 2\sigma
$$

同时可以设置业务绝对阈值，例如最大合理层级为 7 层。满足任一条件的路径进入候选集合：

```text
路径深度 > 7
或
路径深度 > 平均深度 + 2σ
```

| 方法 | 解决什么问题 |
| --- | --- |
| 最大层级阈值 | 找出超过业务允许深度的路径 |
| `mean + 2σ` | 找出相对其他路径明显过深的异常路径 |
| 单链长度 | 找出连续多个节点都只有一个子节点的冗余层级 |
| 路径名称重复率 | 找出父子节点表达相同或包含关系过强的路径 |

#### 解决方法

将异常路径中的节点名称、父子关系、兄弟节点和叶子节点传给模型，判断是否存在：

- 无意义的中间层级；
- 父子节点同名或近义；
- 连续单链；
- 可以合并的中间分类；
- 分类粒度过细。

根据判断生成以下方案：

| 调整方式 | 适用情况 |
| --- | --- |
| 合并中间节点 | 父子节点表达相同、层级没有新增分类含义 |
| 上移子树 | 中间节点仅起转发作用，没有实际分类价值 |
| 重命名节点 | 层级合理但节点名称不能体现分类边界 |
| 保留原结构 | 深度较大但每一层均具有明确业务含义 |

### 4.2 树的宽度

#### P95 阈值判断

设当前节点直接子节点数量为 $k$。当 $k$ 超过所有非叶子节点子节点数量分布的第 95 百分位时，进入候选异常集合：

$$
k > P95
$$

对于“某节点子节点超过 2000 个”这类明显异常情况，还应增加绝对阈值：

```text
k > P95
或
k > 2000
```

绝对阈值用于发现极端大分支，相对阈值用于适应不同规模的分类体系。

#### 相对分支爆炸比

用于判断当前节点是否明显比同级节点更宽：

$$
相对分支爆炸比 = \frac{当前节点的子节点数}{mean(兄弟节点子节点数)}
$$

| 相对分支爆炸比 | 说明 |
| --- | --- |
| `< 2` | 基本正常 |
| `2～5` | 分支偏宽 |
| `> 5` | 明显过宽 |

#### 解决方法

系统不能只根据“子节点多”就直接拆分，而应把父节点名称和具有代表性的子节点名称交给模型，判断子节点是否可以按照稳定业务维度分组。

例如某节点下有 2000 多个子节点，可以尝试按照以下维度拆分：

- 产品用途；
- 原材料；
- 工作原理；
- 行业领域；
- 使用场景；
- 产品形态；
- 技术规格。

模型需要输出完整拆分方案：

```json
{
  "target_node": "人工智能产品",
  "split_dimension": "应用场景",
  "new_groups": [
    {
      "group_name": "工业人工智能产品",
      "children": ["工业视觉系统", "预测性维护系统"]
    },
    {
      "group_name": "医疗人工智能产品",
      "children": ["医学影像辅助系统", "智能诊断系统"]
    }
  ],
  "unassigned_children": [],
  "confidence": 0.86
}
```

系统必须检查：

- 每个原子节点只能分配一次；
- 所有原子节点都必须被分配，或明确列入待人工处理集合；
- 新增分组名称不能与已有兄弟节点重复；
- 拆分后不能形成循环依赖；
- 拆分后最大宽度应明显下降。

### 4.3 树平衡问题

#### 最大/最小子树规模比值

用于判断同一层级下不同分支之间的规模差异：

$$
balance\_ratio = \frac{max(subtree\_size)}{min(subtree\_size)}
$$

| `balance_ratio` | 说明 |
| --- | --- |
| `< 5` | 基本均衡 |
| `5～20` | 存在不均衡 |
| `> 20` | 严重不均衡 |

当最小子树规模为 0 或 1 时，应避免比值被极小分支无限放大，可以使用：

$$
balance\_ratio = \frac{max(subtree\_size)}{max(1, min(subtree\_size))}
$$

#### 信息熵判断

设当前节点有 $n$ 个直接子节点，第 $i$ 个子节点的子树规模占比为 $p_i$：

$$
H = -\sum_i p_i \log p_i
$$

归一化信息熵为：

$$
H_{norm} = \frac{H}{\log n}
$$

| $H_{norm}$ | 说明 |
| --- | --- |
| `> 0.8` | 分布较均衡 |
| `0.5～0.8` | 分布一般 |
| `< 0.5` | 分支规模明显不均衡 |

#### 解决方法

树不均衡本身不一定是错误。系统应先把不均衡分支标记为候选，再结合业务含义判断：

- 大分支是否需要继续拆分；
- 极小分支是否应与其他分支合并；
- 是否存在分类口径不一致；
- 是否只是业务数据天然分布不均。

### 4.4 叶子节点占比

叶子节点占比用于描述分类树的终止节点分布：

$$
叶子节点占比 = \frac{leaf\_nodes}{total\_nodes}
$$

仅使用整棵树的叶子占比很难直接判断问题，因此还应计算某个分支的直接叶子占比：

$$
直接叶子占比 = \frac{当前节点的叶子型直接子节点数}{当前节点的直接子节点数}
$$

| 情况 | 说明 |
| --- | --- |
| 子节点较少且叶子占比高 | 可能是正常终止分类 |
| 子节点很多且叶子占比高 | 可能存在可继续分组的空间 |
| 子节点很多但叶子占比低 | 已经存在较多中间分类，需要结合宽度分析 |

#### 解决方法

只对“子节点数量较多且直接叶子占比异常高”的分支调用模型，判断是否能按照用途、规格或场景进一步分组。不能仅因为叶子占比高就认定为问题。

### 4.5 父节点缺失和结构断裂

根据 `category_id`、`parent_id` 和完整路径检查：

- 父节点 ID 在当前版本中不存在；
- 路径声明的祖先节点与实际父子关系不一致；
- 节点没有父节点但又不是一级分类；
- 路径中出现循环引用；
- 同一节点被挂载到多条结构路径。

#### 解决方法

| 问题 | 调整方式 |
| --- | --- |
| 父节点记录缺失但路径可恢复 | 根据路径补充父节点或恢复父子关系 |
| 节点实际应属于其他已有分支 | 生成 `move_node` 建议 |
| 中间分类在数据中完全缺失 | 生成 `add_node` 建议，并将相关子节点挂载到新节点 |
| 循环依赖 | 阻止执行并要求人工处理 |
| 重复挂载 | 保留主路径，其他路径删除或改为关联关系 |

## 5. 内容优化方案

内容优化主要处理仅依赖树结构无法准确判断的问题，包括冗余节点、重复挂载、父子关系不合理、命名不规范、粒度不一致和同义词问题。

### 5.1 父子节点不匹配

父子节点不匹配用于判断父节点和子节点在语义领域上是否一致，避免出现跨领域挂载错误。

例如：

```text
父节点：工业产品类
子节点：薯类
```

“薯类”属于农业或食品领域，不适合挂载在“工业产品类”下。

#### 判断思路

首先根据一级分类形成领域关键词词库：

```python
{
  "农业": ["农", "粮", "谷", "米", "麦", "薯", "豆", "果", "蔬菜", "畜", "禽"],
  "工业": ["工业", "机械", "设备", "机床", "泵", "阀", "钢", "化工"],
  "食品": ["食品", "饮料", "肉", "奶", "糖", "酒", "调味"],
  "服装": ["服装", "衣", "裤", "鞋", "帽", "纺织"],
  "电子": ["电子", "手机", "电脑", "数码", "芯片", "电路"]
}
```

规则先检查明显的领域冲突，再结合以下上下文调用模型复核：

- 当前父节点和祖先路径；
- 子节点名称和同义词；
- 子节点的兄弟节点；
- 子节点的直接子节点；
- 向量检索得到的相似节点及其路径。

| 判断情况 | 处理方式 |
| --- | --- |
| 父子领域一致 | 标记为正常 |
| 出现明显跨领域关键词 | 进入模型复核 |
| 父节点含义过于宽泛 | 结合祖先路径和兄弟节点判断 |
| 模型证据不足 | 标记为“不确定”，不生成自动执行动作 |

#### 解决方法

模型确认父子关系错误后，优先从现有节点中寻找更合理的父节点，生成 `move_node` 建议；如果没有合适父节点，则生成 `add_node` 建议。调整前必须检查目标父节点是否会造成循环依赖或名称冲突。

### 5.2 冗余节点和重复节点

冗余节点包括：

- 同一父节点下名称完全相同的节点；
- 不同路径下名称相同且语义相同的节点；
- 名称不同但含义高度接近的近义节点；
- 父子节点同名或含义基本相同；
- 只起转发作用、没有分类含义的中间节点。

#### 候选发现方法

| 方法 | 适用问题 |
| --- | --- |
| 名称标准化后精确匹配 | 完全重复名称 |
| 编辑距离、Jaccard 相似度 | 字面表达相近 |
| Embedding 向量相似度 | 语义近似但名称不同 |
| 父子名称包含关系 | 父子层级冗余 |
| 同义词交集 | 不同节点共享大量同义词 |

节点名称标准化可包括：

- 去除首尾空格；
- 统一全角和半角字符；
- 统一大小写；
- 去除无意义标点；
- 统一常见简称；
- 对括号内容进行单独比较。

#### 解决方法

模型判断两个节点是否真正表达同一分类，并输出：

- 建议保留的主节点；
- 建议合并的从节点；
- 两个节点原有子节点如何迁移；
- 同义词如何合并和去重；
- 是否存在 ID 被其他记录引用的风险。

确认后执行 `merge_node`。合并不能简单删除从节点，必须迁移子节点、更新路径、处理同义词并保留旧 ID 到新 ID 的映射记录。

### 5.3 重复挂载节点

重复挂载是指同一个分类对象出现在多个不合理路径下。例如同一个 `category_id` 被多个父节点引用，或不同 ID 的节点名称、同义词和子树高度一致，实际表示同一对象。

#### 判断方法

1. 检查同一 `category_id` 是否对应多个 `parent_id`；
2. 检查同一节点是否存在多条完整路径；
3. 对名称、同义词和子树结构均高度一致的节点进行相似判断；
4. 区分“允许的多维分类”与“错误重复挂载”。

#### 解决方法

| 情况 | 处理方式 |
| --- | --- |
| 数据错误导致同一 ID 多父节点 | 选择正确父节点，移除错误挂载 |
| 两个不同 ID 实际为同一分类 | 合并节点并保留 ID 映射 |
| 业务允许多维归类 | 保留一个主分类路径，其他关系改为关联标签 |
| 无法确定主路径 | 提交人工审核，不自动删除 |

### 5.4 命名不规范

命名规范性判断用于发现名称过长、过短、格式混乱、混入型号、空值或表达模糊的问题。

| 问题类型 | 判断方式 | 说明 |
| --- | --- | --- |
| 名称过长 | 字符长度超过阈值 | 可能混入描述性语句 |
| 名称过短 | 长度过短且不属于允许简称 | 可能表达不清 |
| 特殊符号异常 | 包含大量特殊符号 | 影响展示和检索 |
| 数字型号混入 | 正则识别型号、规格和商品编号 | 可能把具体商品作为分类节点 |
| 空值节点 | 名称为空或仅包含空白字符 | 数据缺失 |
| 模糊名称 | 出现“其他”“综合”“通用”等词 | 分类边界不清晰 |

例如：

```text
海尔BCD-216冰箱
手机/电脑&数码类
其他
NULL
```

#### 解决方法

规则可以直接识别空值、特殊符号和明显型号。对于语义模糊或需要保留行业术语的节点，再调用模型生成推荐名称。推荐名称必须结合父节点、兄弟节点和子节点，不能仅对单个名称进行改写。

### 5.5 粒度不一致和父子关系倒挂

粒度不一致是指同一父节点下的子节点抽象程度不统一，例如：

```text
手机
笔记本电脑
华为 Mate 70 Pro
智能穿戴设备
```

其中“华为 Mate 70 Pro”是具体型号，粒度明显比其他分类节点更细。

父子关系倒挂是指父节点比子节点更具体，或者子节点概念范围反而更宽。例如：

```text
父节点：苹果手机
子节点：智能手机
```

#### 解决方法

以兄弟节点组和完整父子路径为分析单位调用模型，判断：

- 各兄弟节点是否属于同一抽象层次；
- 是否混入品牌、型号、规格或实例；
- 父节点是否能够完整包含子节点；
- 是否需要上移、下移、重命名或新建中间分类。

### 5.6 抽样调查

部分语义问题无法通过规则稳定筛选，因此可以从一开始采用抽样方式发现隐藏问题。

整体流程：

```text
全部节点
  ↓
按一级分类进行分层抽样
  ↓
补充大分支、深路径和高风险节点样本
  ↓
构造节点、父子节点对或兄弟节点组
  ↓
批量调用模型判断
  ↓
生成问题说明和修改建议
```

| 问题类型 | 抽样对象 |
| --- | --- |
| 近义节点 | 同层节点组或向量相似节点组 |
| 模糊节点 | 单个节点名称及上下文 |
| 父子关系倒挂 | 父子节点对 |
| 粒度不一致 | 兄弟节点组 |
| 路径冗余 | 完整深层路径 |

#### 抽样策略

| 抽样方式 | 说明 |
| --- | --- |
| 随机抽样 | 从全部节点中等概率抽取，用于发现规则未覆盖的问题 |
| 分层抽样 | 每个一级分类都抽取一定数量，防止样本集中在大类 |
| 高风险节点补充 | 将规则命中的可疑节点加入样本，但不代替随机抽样 |
| 兄弟节点成组抽样 | 适合判断粒度和分类口径 |
| 深层路径抽样 | 适合发现层级冗余和父子倒挂 |
| 大分支抽样 | 重点分析子节点超过阈值的节点 |

随机抽样和规则筛查可以同时存在：随机抽样用于发现未知问题，规则筛查用于提高已知风险问题的检查效率。

### 5.7 同义词问题

同义词字段用于提高分类节点的检索召回能力，但长期维护中可能出现同义词缺失、重复、冲突或语义污染。

#### 同义词问题类型

| 问题类型 | 说明 | 示例 |
| --- | --- | --- |
| 同义词缺失 | 重要节点没有常见别名或简称 | “新能源汽车”缺少“新能源车” |
| 同义词重复 | 同一节点的同义词列表中存在重复项 | `新能源车, 新能源车` |
| 同义词等于主名称 | 同义词与节点名称完全一致 | 节点“手机”，同义词“手机” |
| 同义词污染 | 同义词与当前节点语义无关 | 水果“苹果”的同义词包含“iPhone” |
| 跨节点冲突 | 同一个同义词被多个不相关节点使用 | “苹果”同时作为水果和手机节点同义词 |
| 上下位词误作同义词 | 把父类、子类或具体实例当作同义词 | “水果”的同义词包含“苹果” |
| 格式不统一 | 分隔符、空格、大小写或全半角不统一 | `AI； ai,人工智能` |

#### 规则检测

1. 对同义词字符串进行分隔、清洗和标准化；
2. 删除空值、重复值和与主名称完全相同的词；
3. 建立“同义词 → 节点”倒排索引，发现跨节点冲突；
4. 使用向量相似度检查同义词与主节点名称、完整路径的语义一致性；
5. 检查同义词是否与父节点、子节点或具体型号高度相似，识别上下位词误用。

可定义同义词语义一致度：

$$
synonym\_consistency = \frac{有效同义词数量}{同义词总数量}
$$

#### 解决方法

| 问题 | 调整方式 |
| --- | --- |
| 重复、空值或与主名称相同 | 规则直接清理 |
| 格式不统一 | 统一分隔符、大小写和空格 |
| 明显无关同义词 | 生成 `clean_synonym` 建议 |
| 同义词缺失 | 模型结合行业常用表达生成候选，人工确认后补充 |
| 跨节点冲突 | 结合完整路径判断应归属哪个节点，必要时两个节点均保留但增加消歧说明 |
| 上下位词误用 | 从同义词列表移除，并保留为父子分类关系 |

同义词自动补充属于中风险操作，不能直接把模型生成结果批量写入正式版本，应经过人工审核。

## 6. 自动调整与安全执行

系统发现问题后，需要把自然语言建议转换为结构化动作。建议记录至少包含：

```json
{
  "issue_id": 101,
  "action_type": "move_node",
  "target_node_id": "26459",
  "old_parent_id": "1200",
  "new_parent_id": "8000",
  "reason": "当前节点属于农业领域，不应挂载在工业产品类下。",
  "risk_level": "high",
  "confidence": 0.94,
  "need_human_review": true
}
```

### 6.1 支持的动作类型

| 动作类型 | 说明 | 默认风险 |
| --- | --- | --- |
| `rename_node` | 修改节点名称 | 中 |
| `move_node` | 移动节点到新的父节点 | 高 |
| `add_node` | 增加缺失父节点或中间分类 | 中 |
| `merge_node` | 合并冗余节点 | 高 |
| `split_subtree` | 将过宽分支拆分为多个中间分类 | 高 |
| `clean_synonym` | 删除或修改错误同义词 | 中 |
| `add_synonym` | 补充同义词 | 中 |
| `delete_leaf_node` | 删除无效叶子节点 | 高 |
| `mark_as_valid` | 将规则候选标记为合理结构 | 低 |

### 6.2 执行前校验

所有动作必须先在内存中的版本快照上模拟，校验：

- 目标节点和目标父节点是否存在；
- 是否形成循环依赖；
- 是否出现重复 ID；
- 是否出现同级重名；
- 是否导致路径断裂；
- 子节点和同义词是否完整迁移；
- 拆分方案是否覆盖全部原子节点；
- 动作执行后问题指标是否改善。

### 6.3 人工审核策略

| 风险等级 | 处理方式 |
| --- | --- |
| 低风险 | 可以批量确认，例如格式统一、删除重复同义词 |
| 中风险 | 默认逐条审核，例如重命名、补充同义词 |
| 高风险 | 必须查看修改前后预览后确认，例如移动、合并、拆分和删除 |

审核通过后由程序执行动作，不允许模型直接操作数据库。

## 7. 体系评价方案

体系评价用于回答两个问题：

1. 当前分类体系质量如何；
2. 执行调整后，体系质量是否真正提高。

评价采用 100 分制，至少包含结构合理性、内容准确性和同义词质量三个维度。各指标权重可以根据业务需要配置。

### 7.1 结构合理性评分

结构合理性可以由以下指标组成：

| 指标 | 含义 | 建议权重 |
| --- | --- | ---: |
| 深度合理率 | 未超过深度阈值的路径比例 | 25% |
| 宽度合理率 | 未超过宽度阈值的非叶子节点比例 | 25% |
| 结构完整率 | 父节点存在且路径一致的节点比例 | 25% |
| 分支均衡度 | 根据归一化信息熵和规模比计算 | 15% |
| 冗余层级控制 | 不存在连续单链和父子同名的路径比例 | 10% |

深度合理率：

$$
R_{depth} = 1 - \frac{异常深路径数}{全部根到叶路径数}
$$

宽度合理率：

$$
R_{width} = 1 - \frac{异常宽节点数}{全部非叶子节点数}
$$

结构完整率：

$$
R_{complete} = 1 - \frac{父节点缺失数 + 路径断裂数 + 循环节点数}{全部节点数}
$$

结构评分：

$$
S_{structure} = 100 \times (0.25R_{depth} + 0.25R_{width} + 0.25R_{complete} + 0.15R_{balance} + 0.10R_{redundancy})
$$

### 7.2 内容准确性评分

内容准确性包括：

- 父子语义关系准确率；
- 重复或冗余节点控制；
- 重复挂载控制；
- 节点命名规范率；
- 同层粒度一致率。

对于规则可以全量检查的问题，分母使用全部相关节点；对于必须抽样或模型判断的问题，分母使用本轮实际评估样本数，并明确标注样本量。

父子关系准确率：

$$
R_{parent} = 1 - \frac{确认错误的父子节点对数量}{本轮评估的父子节点对数量}
$$

节点唯一性：

$$
R_{unique} = 1 - \frac{确认冗余或重复挂载节点数}{全部节点数}
$$

命名规范率：

$$
R_{naming} = 1 - \frac{命名不规范节点数}{全部节点数}
$$

内容评分：

$$
S_{content} = 100 \times (0.35R_{parent} + 0.25R_{unique} + 0.20R_{naming} + 0.20R_{granularity})
$$

需要注意：如果内容评价来自抽样，报告必须同时显示“评价样本数”和“抽样方式”，不能把抽样准确率直接描述为全量准确率。

### 7.3 同义词覆盖率和质量评分

同义词评价不能只检查字段是否非空，还要同时考虑覆盖率和有效性。

同义词覆盖率：

$$
R_{coverage} = \frac{具有至少一个有效同义词的节点数}{需要配置同义词的节点数}
$$

同义词有效率：

$$
R_{valid} = \frac{通过重复、冲突和语义一致性检查的同义词数量}{同义词总数量}
$$

同义词评分：

$$
S_{synonym} = 100 \times (0.6R_{coverage} + 0.4R_{valid})
$$

对于不需要同义词的内部结构节点，可以通过规则排除，避免强制所有节点都配置同义词。

### 7.4 综合评分

综合评分可以定义为：

$$
S_{total} = 0.4S_{structure} + 0.4S_{content} + 0.2S_{synonym}
$$

| 分数区间 | 评价 |
| --- | --- |
| `90～100` | 体系质量优秀 |
| `80～89` | 体系质量较好，存在少量问题 |
| `70～79` | 体系基本可用，需要持续优化 |
| `60～69` | 问题较多，建议重点治理 |
| `< 60` | 结构或内容质量较差，需要系统性调整 |

权重和分数区间应配置化，不能写死在模型 Prompt 中。

### 7.5 维护前后效果评价

执行动作后，在旧版本和新版本上使用完全相同的指标版本重新计算：

$$
\Delta S = S_{new} - S_{old}
$$

报告中应展示：

```text
结构评分：72.4 → 81.6（+9.2）
内容评分：68.0 → 78.5（+10.5）
同义词评分：61.3 → 70.1（+8.8）
综合评分：69.6 → 79.3（+9.7）

已解决问题：38
仍未解决问题：12
新引入问题：2
```

如果新版本引入的问题多于解决的问题，或综合评分下降，系统应标记该版本需要复核，而不能仅因为动作执行成功就判断维护成功。

## 8. 版本管理方案

版本管理用于记录标准产品体系每次维护的输入、问题、建议、审核结果、执行动作和最终结果。

### 8.1 版本原则

1. 原始 Excel 导入后创建初始版本 `v1.0`；
2. 历史版本不可直接覆盖；
3. 每次执行一批审核通过的动作后创建新版本，例如 `v1.1`；
4. 新版本保存父版本 ID，形成版本链；
5. 诊断问题、建议、审核和执行日志都绑定具体版本；
6. 用户可以从任意历史版本重新开始维护；
7. 回滚不是删除新版本，而是以历史版本为基础生成新的恢复版本。

### 8.2 版本记录内容

每个版本至少记录：

| 字段 | 说明 |
| --- | --- |
| `version_id` | 版本唯一标识 |
| `version_no` | 版本号，例如 `v1.0`、`v1.1` |
| `parent_version_id` | 父版本 ID |
| `source_file_id` | 原始 Excel 文件 ID |
| `workflow_id` | 产生该版本的工作流 ID |
| `created_time` | 创建时间 |
| `operator` | 操作人 |
| `description` | 版本说明 |
| `quality_score` | 综合质量评分 |
| `node_count` | 节点总数 |
| `issue_count` | 当前诊断问题数 |
| `status` | 正常、待复核、已归档等 |

### 8.3 维护过程记录

系统应记录完整证据链：

```text
版本 v1.0
  ↓
诊断运行 run-001
  ↓
发现问题 issue-101
  ↓
生成建议 suggestion-201
  ↓
人工审核 review-301
  ↓
执行动作 action-401
  ↓
生成版本 v1.1
  ↓
验证运行 verify-002
```

这样可以回答：

- 某个节点为什么被移动；
- 调整建议由哪个问题产生；
- 谁批准了该动作；
- 动作何时执行；
- 新版本质量是否提高。

### 8.4 版本差异分析

两个版本之间可以比较：

- 新增节点；
- 删除节点；
- 重命名节点；
- 移动节点；
- 合并节点；
- 拆分分支；
- 同义词新增、删除和修改；
- 层级深度变化；
- 最大分支宽度变化；
- 各类诊断问题数量变化；
- 质量评分变化。

版本差异示例：

```text
版本：v1.0 → v1.1

节点变化：
- 新增节点：12
- 删除节点：3
- 重命名节点：8
- 移动节点：15
- 合并节点：4组
- 同义词修改：37项

结构变化：
- 最大深度：10 → 8
- 最大直接子节点数：2645 → 412
- 父节点缺失数：44 → 3

质量变化：
- 综合评分：69.6 → 79.3
```

### 8.5 月度和年度差异分析

系统可以按照时间范围汇总版本变化：

| 分析类型 | 内容 |
| --- | --- |
| 月度分析 | 本月新增、删除、移动、重命名节点数量，问题解决率和质量变化 |
| 季度分析 | 各一级分类的调整次数、问题集中领域和评分趋势 |
| 年度分析 | 年初与年末版本差异、全年累计动作、体系规模变化和治理效果 |

例如年度分析：

```text
2026 年度版本变化

年初版本：v1.0
年末版本：v3.4
节点数量：21,090 → 21,376
累计新增：524
累计删除：238
累计移动：416
累计合并：73组
父子关系错误：126 → 18
综合评分：69.6 → 88.2
```

### 8.6 回滚和继续维护

用户可以选择某个历史版本作为新的维护基础：

```text
v1.0 → v1.1 → v1.2
          ↓
       发现结果不理想
          ↓
以 v1.0 为基础生成 v1.3-recovery
```

系统不应物理删除 `v1.1` 和 `v1.2`，以保证历史过程可追溯。

系统页面不能只展示“工作流完成”，还应让用户看出具体结果。

### 9.1 诊断概览


| 问题类型 | 节点 | 完整路径 | 风险 | 置信度 | 来源 |
| --- | --- | --- | --- | ---: | --- |
| 父子关系错误 | 薯类 | 产业 > 工业产品类 > 薯类 | 高 | 94% | 规则 + 模型 |

### 9.3 问题详情

- 当前节点和完整路径；
- 父节点、子节点和兄弟节点；

### 9.4 最终报告

最终报告至少包括：

2. 分类体系基本统计；
3. 结构问题分析；
4. 内容问题分析；
5. 同义词问题分析；
6. 体系质量评分；
7. 调整建议和审核结果；
8. 实际执行动作；
9. 新旧版本差异；
10. 维护前后质量变化；
11. 未解决问题和下一步优化建议。

## 10. 方案总结

本方案将标准产品体系维护划分为四个相互衔接的部分：

```text
结构维护
检测深度、宽度、平衡性、断链和冗余层级
        ↓
内容优化
检测冗余节点、重复挂载、父子关系、命名和同义词问题
        ↓
体系评价
计算结构、内容和同义词质量评分，比较调整前后变化
        ↓
版本管理
记录问题、建议、审核、动作和版本差异，支持持续维护
```

规则负责稳定、全量的确定性检查；模型负责需要语义理解的判断和调整方案；人工审核保证高风险修改可控；程序负责合法性校验和实际执行；版本管理保证每次变化都可追溯、可比较和可恢复。

最终形成以下维护闭环：

```text
发现问题 → 解释问题 → 生成方案 → 人工确认 → 安全执行
→ 创建新版本 → 重新评价 → 继续优化
```
