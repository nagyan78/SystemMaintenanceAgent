# 修复方案：可迭代审核 + 版本续跑（循环节点设计）

> 目标：让"分类体系维护"从「一次性单程管道」升级为「可多轮迭代的维护会话」，
> 解决三个实测缺口：①审核只能一次；②重跑从原始 v1.0 重来、已修的 v2 不被复用；
> ③版本是静态快照、无自动复检。
> 约束：**不改动已有的确定性脊与规则路由**（结构诊断保持规则、LLM 只在内容/建议/报告节点）。

---

## 1. 根因（对照代码）

| 现象 | 根因 | 代码位置 |
|------|------|----------|
| 审核只能一次 | 全图仅 `wait_human_review_node` 一个 `interrupt()`，恢复后直奔 END，无回路；`resume_workflow` 仅在 `status=="waiting_review"` 可调用，跑完即终态 | graph.py: `route_after_review`；workflows.py:161 |
| 重跑从原始 v1.0 重来 | `start` 只收 `file_id`；`create_initial_version` 复用已存在的 v1.0，使新一轮 `current_version_id` 固定为 v1.0 | workflows.py:23-24；version_service.py:18-29 |
| v2 被丢弃 | `save_new_version_node` 以 `current_version_id`（=v1.0）为 base，而非该文件最新版本 | nodes.py:310 |
| 无跨轮待办 | 建议状态机只有 pending/approved/rejected/edited/executed/failed，无 "deferred"；驳回项不带入下一轮 | suggestion_repo / review_service |
| 版本终态化 | 缺 session/round 概念，版本链不被工作流消费 | 无 |

> 补充不一致点：`review_service.execute_approved_actions` 用 `get_latest_for_file` 作 base（review_service.py:120），
> 而 workflow 路径的 `save_new_version_node` 用 `current_version_id`。**两者应统一为"最新版本"**。

---

## 2. 设计总览：两类循环节点

LangGraph 原生支持有环图（cycle），且 `interrupt()` 在环内每经过一次都会产生新 checkpoint。
我们引入**两种互补的循环机制**：

### 2.1 循环 A — 单 run 内「多次审核」（intra-run review loop）
执行完本批已批准项后，若**仍有待审建议且未达上限**，路由回 `wait_human_review_node` 再次中断，
由用户对剩余项继续审核。每次回到该节点都会把 `task.status` 重设为 `waiting_review`，
因此既有的 resume 机制天然复用。

### 2.2 循环 B — 跨轮「从版本续跑」（inter-run round loop）
新增可重入入口 `bootstrap_node`：按 `mode` 分支——
- `from_file`：parse → build → save_initial_version（v1.0，session 起始）
- `from_version`：直接 `load_version(version_id)` 设为 `current_version_id`（跳过解析）

两种模式汇合后走同一套诊断→建议→审核流程。几天后用户点「从此版本续跑」，
即以 `from_version` 模式重新进入同一张图，实现真正的"下一轮"。

> 结论：**可以设计循环节点，而且应该用两个环**——A 解决"一次审不完"，B 解决"过几天再修别的"。
> 不需要"自动定时复检"（那会无谓烧 API）；用户主动续跑 + 增量诊断更可控。

## 3. 新拓扑（见附图）

```
            ┌─ from_file: parse→build→save_initial(v1.0)
bootstrap ──┤
            └─ from_version(continue): load_version(vN)   ← 用户点「续跑」

  bootstrap → index_vector(幂等跳过) → diagnosis_planning → structure_diagnosis
           → content_diagnosis → generate_suggestion(合并上一轮 deferred)
           → wait_human_review(LOOP-A 入口)
                │
           route_after_review
             ├─ 整批拒绝 → generate_report → END
             └─ 有批准  → validate_action → execute_action
                             → route_after_round
                                  ├─ 仍有待审 & 未达 MAX → 回 wait_human_review (LOOP-A)
                                  └─ 否则 → save_new_version(base=最新版本) → report → END
```

## 4. State 改动（`backend/app/agents/states.py`）

现有 `base_version_id / current_version_id / new_version_id` 保留，新增：

```python
session_id: str | None = None          # 维护会话，跨轮恒定
round_index: int = 0                   # 第几轮（from_file 起 0）
mode: Literal["from_file", "from_version"] = "from_file"
review_iteration: int = 0             # 本轮内已回环审核次数
pending_suggestion_ids: list[int] = []# 本轮尚未决定的建议
deferred_suggestion_ids: list[int] = []  # 跨轮待办
MAX_REVIEW_ITERATIONS: int = 3         # 类常量，防环失控
```

---

## 5. 节点改动清单

### 5.1 新增 `bootstrap_node`（graph.py / nodes.py）
读取 `state.mode`：
- `from_file` → 调 `parse_excel_node → build_tree_node → save_initial_version_node`，
  并写入 `session_id = new_uuid()`、`round_index = 0`。
- `from_version` → 调 **新增 `load_version_node`**：`VersionService.get_version(version_id)`
  设为 `current_version_id`，从目标版本读出 `session_id` 并 `round_index = prev+1`，
  **跳过解析与建树**（节点已落库）。

### 5.2 `wait_human_review_node`（nodes.py:233）
- 每次进入都 `task_repo.update_status(task_id, "waiting_review")`（保证循环 A 可 resume）。
- 决策 payload 改为**逐条**：`approve / reject / edit / defer`。
- `defer` 的建议标记 `status="deferred"` + `session_id` + `round_index`，且不立即执行。

### 5.3 新增 `route_after_round`（graph.py）
替代"执行后直接 save_version"的硬连线：
```python
def route_after_round(state):
    if state.pending_suggestion_ids and state.review_iteration < MAX_REVIEW_ITERATIONS:
        return "wait_human_review"   # LOOP-A
    return "save_new_version"
```
`wait_human_review_node` 进入时 `review_iteration += 1`。

### 5.4 `generate_suggestion_node`（nodes.py:212）—— 合并 deferred
在生成本轮建议后，追加查询 `list_suggestions(status="deferred", session_id=...)`，
把上一轮暂缓的项以"此前您暂缓的建议"标注重新呈现为 pending。

### 5.5 `save_new_version_node`（nodes.py:301）—— **修正 base bug**
```python
# 旧：base_version_id = state.current_version_id   # 常是 v1.0，丢 v2
# 新：以该文件最新版本为 base
latest = VersionRepository(settings).get_latest_for_file(state.file_id)
base_version_id = int(latest["id"])
```
并写入 `session_id`、`round_index`，版本号按 `_next_version_no` 自然递增（v1.0→v1.1→v1.2…）。

### 5.6 `review_service.apply_workflow_decision`（review_service.py）
区分 `rejected`（明确放弃，不跨轮）与 `deferred`（暂缓，下一轮复现）。

## 7. API 改动（`backend/app/api/workflows.py`）

- `StartWorkflowRequest`（workflows.py:23）：加 `version_id: int | None = None`。
  有值即 `mode="from_version"`。
- 新增 `POST /api/workflows/taxonomy/continue`：`{version_id, full_rediagnose?: bool}`，

- `GET /api/versions`：支持 `?session_id=` 过滤；新增 `GET /api/versions/{id}/deferred` 列出跨轮待办。

## 9. 关键风险与对策

| 风险 | 对策 |
|------|------|
| 无限循环 | `MAX_REVIEW_ITERATIONS=3` 封顶；且每次回环都需人工 resume，无法无人自转 |
| API 成本 | `from_version` 模式支持 `full_rediagnose=false` 的**增量诊断**（仅查变更节点/deferred 候选集）；向量索引 `index_version` 已幂等跳过 |
| 版本链错乱 | 修正 `save_new_version` base = 最新版本（§5.5），彻底解决 v2 被丢弃 |
| 中断一致性 | 每次进入 `wait_human_review` 重设 `waiting_review`；SQLite checkpointer 已支持同 thread 多 checkpoint |
| 架构评审红线 | 结构诊断保持规则、确定性分支保持规则路由；循环只动"审核/建议/版本"编排层 |

## 10. 落地顺序（建议 3 个 PR）

**PR1（最小可用，先止血"重跑从 v1.0"）**
- state/model 加 session/round/mode；`bootstrap_node` + `from_version` 模式 + `load_version_node`；
- continue 端点；**修正 `save_new_version` base=最新版本**；
- 测试：bootstrap 两模式、base 解析为最新而非 v1.0。

**PR2（单 run 内多轮审核）**
- `route_after_round` 循环 + `wait_human_review` 重入 + `review_iteration`；
- ReviewView「继续审核」UX；
- 测试：审核循环在 MAX 处收敛、resume 在环中可用。

**PR3（跨轮 deferred + 增量诊断 + 前端续跑）**
- `deferred` 状态 + `generate_suggestion` 合并复现；
- `from_version` 增量诊断选项；
- VersionsView 续跑按钮 + 缺陷待办列表；
- 测试：deferred 下一轮复现、版本链 v1.0→v1.1→v1.2 正确、quality_score 单调改善。
