# R1 可信诊断与完整结果执行计划

> 状态：CURRENT / IN PROGRESS  
> 路线来源：[当前开发路线图](../00_当前状态/当前开发路线图.md)

## 目标

让用户上传产品标准体系后，获得覆盖范围明确、可定位、可解释、可追溯的完整问题清单，并避免因模型 Token 预算耗尽导致整次任务无报告。

## 实施范围

1. 补齐 taxonomy 与 diagnosis 查询 API，支持树浏览、节点搜索、问题筛选和证据查看。
2. 建立“全量规则轻筛查 → 候选召回 → 重点 Agent 深诊断”的覆盖漏斗。
3. 记录扫描节点数、候选数、深诊断数、问题数、跳过数及未诊断原因。
4. 让诊断计划真实约束节点范围、问题类型、批次和 Token 预算。
5. 模型预算不足时保留规则结果和已完成批次，任务以“部分完成”生成报告，不因 AI 增强失败丢失全部结果。
6. 统一中文问题分类与展示名称，并保留稳定的内部问题代码。
7. 问题详情关联节点、完整路径、规则或语义证据、置信度、来源和 workflow run。
8. 修复失败状态、报告写入和数据库外键之间的证据链一致性。

## 非目标

- 不在本阶段建设分布式多 Agent 平台。
- 不让 LLM 替代确定性规则或直接修改数据。
- 不以增加模型调用量代替覆盖率设计。

## 交付顺序

1. 定义 run 级诊断结果 schema、覆盖率指标和部分完成状态。
2. 补齐 taxonomy/diagnosis 查询能力。
3. 实现全量轻筛查、候选召回、分批深诊断和断点结果保留。
4. 使计划范围、问题类型和预算约束进入实际执行。
5. 将中文问题分类、节点路径和证据接入前端及报告。
6. 使用固定 Excel 与已知问题清单执行端到端验收。

## 验收标准

- 固定样例的已知问题召回率可以计算并重复验证。
- 每个报告问题都能定位到节点与证据；内部代码和中文名称同时可追溯。
- Token 预算耗尽时，任务不会只显示失败百分比；至少生成规则诊断、已完成 AI 结果、覆盖率和中断原因。
- 重试或续跑不会重复写入同一批问题，也不会触发外键错误。
- 诊断数量、评分和报告内容均来自真实执行结果，不硬编码。
