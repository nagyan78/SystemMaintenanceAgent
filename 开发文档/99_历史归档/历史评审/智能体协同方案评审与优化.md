# 产品标准体系维护智能体 — 代理协同方案评审与优化建议

> 评审视角：调度达（AgentsOrchestrator）｜基于：`graph.py` / `nodes.py` / `states.py` / `checkpoints.py` / `dev-doc/10_*` / `架构评审报告.md`
> 结论先行：**设计方向正确、落地扎实，不是"工作流 vs 智能体"的方向错误，而是"智能体能力层"在个别环节还能更深。** 整体不必推倒重来，按下面优先级做增量增强即可。

---

## 一、先说优点（这些都是对的，请保留）

1. **"确定性脊 + 内嵌智能体"的混合架构是正确选择。**
   你们用一条固定 DAG 当编排骨架（`parse → build → save → index → structure_diagnosis → planning → content_diagnosis → suggestion → review → validate → execute → version → report`），只在 `diagnosis_planning / content_diagnosis / generate_suggestion / generate_report` 四个节点里嵌入 LLM 驱动的 agent loop。这比"把多个 agent 丢进自由聊天互相调用"稳得多——后者在 2 万节点规模下会失控、不可复现、无法审计。**你们现在的形态正是业界推荐的 Plan-and-Execute / Supervisor 模式，不用改。**

2. **节点足够"薄"。** `nodes.py` 里每个节点只做三件事：调一个 service、写 progress event、返回 `state.model_copy(update=...)`。没有在节点里写 SQL、拼 prompt、直接碰 Qdrant。分层（agent / service / repository / tool）清晰，可单测。

3. **"规则优先"落实得到位。** 结构诊断、动作校验、执行、版本保存全是纯规则/程序逻辑，只有需要语义判断的地方才交给 LLM。符合 PRD "规则能解决的不交给大模型"。

4. **人工审核用 `interrupt()` 而非轮询。** 高风险动作前真正暂停、等 `Command(resume=...)` 恢复，是标准且正确的 HITL 实现。

5. **持久化 checkpointer 已到位。** `checkpoints.py` 用常驻连接的 `SqliteSaver`，满足"服务重启后同 thread_id 可恢复"，不是 `InMemorySaver` 的玩具版。

6. **`content_diagnosis` 有硬上限保护。** `plan.estimated_candidates = min(..., 50)` 把 ReAct 循环的 API 调用量锁死，避免 2 万节点全量扫描烧钱。这是很好的 guardrail。

## 二、值得优化的点（按优先级）

### P0-1｜Agent loop 是"黑盒单步"：不可观测、不可中途续跑
**问题：** `content_diagnosis_node` 调用 `ContentDiagnosisAgent.run()`，内部对最多 50 个候选逐个跑 ReAct 循环——但整个循环发生在**一个 graph 节点步**内部。后果有三：
- 跑到第 48 个候选时 API 超时/报错，整个节点失败，`node_guard` 把状态置 `failed`，**之前 47 个的成果全丢，必须从头重跑**；
- 你们 M5 的"展示 Thought-Action-Observation"目标，目前只有节点级 `workflow_step` 事件（`events.py` 的 `map_workflow_event`），**agent 内部每次 tool call 没有事件**，前端看不到推理过程；
- 单步执行时间过长，可能触碰 SSE/HTTP 的超时上限。

**优化：** 把"逐候选诊断"建模成 LangGraph 的 `Send` API / map-reduce——每个候选是一次独立子调用，或至少在 agent 内部每完成一次 tool call 就 `record_event(event_type="agent_tool_call", payload={thought, action, observation})`。这样：
- 候选级可观测、可断点续跑（失败只重跑剩余候选）；
- 前端能真实呈现 ReAct 链；
- 顺便解锁下一步的并行化。

### P0-2｜候选诊断是串行的，没用上并行
**问题：** 50 个候选之间完全独立，却串行跑，墙钟时间 = 50 × 单次 ReAct 耗时。
**优化：** 在 P0-1 的基础上做**有界并发的 fan-out**（map-reduce，并发数按 DeepSeek 速率限制设上限 + 退避）。2 万节点的真实数据下，这一步能把内容诊断从"分钟级"压到"秒级~十秒级"。

### P1-1｜缺一条"校验失败 → 建议重生成"的回流闭环
**问题：** `validate_action_node` 一旦失败，直接 `raise` → 路由到 `generate_report_node`（报告里标注失败），**没有让 LLM 看到失败原因去自我修正**。架构评审报告 §3.6 其实已经点过这个机会，但当前 `graph.py` 的 `route_after_validate` 只有 `failed → report` 一条路。
**优化：** 校验失败时，把失败原因（哪些动作非法、为什么）写回 `state.review_payload`，路由回 `generate_suggestion_node` 重新生成（带反馈的 Reflexion 式循环），设最大重试次数（如 2 次）封顶后若仍失败再进报告。这是让"多代理协同"真正闭环的关键一笔。

### P1-2｜诊断规划是"单次静态计划"
**问题：** `diagnosis_planning_node` 一次 LLM 调用产出 `DiagnosisPlan` 后就固定了，对 2 万节点来说粒度太粗。
**优化：** 升级成**自适应/滚动规划**（Plan-and-Execute）：先规划 → 诊断一小批 → 用实际命中率回看（某些子树没问题就缩圈，某些子树问题密集就扩圈）→ 再规划。或者至少让 `content_diagnosis` 在运行中能更新 `state.diagnosis_plan`，把"看到的情况"反哺给规划。

### P2-1｜单一模型、无路由、无降级
**问题：** 规划、诊断、建议、报告全部用同一个 DeepSeek chat 模型。不同任务复杂度/成本差异很大（规划用便宜快模型即可，诊断推理才需要强模型），且 DeepSeek 一旦不可用整个链路挂掉。
**优化：** 引入一层**模型路由配置**（任务 → 模型档位）可选**降级链**（主用 DeepSeek，备用另一家 OpenAI 兼容端点）。对课程演示的健壮性很有价值。

### P2-2｜智能体间缺"共享工具缓存 / 工作记忆"
**问题：** 诊断 agent 和建议 agent 都会调 `get_node_detail` / `search_similar_nodes`，同一节点可能被两边重复查，Qdrant 语义召回是贵操作。目前没有跨 agent 的结果缓存。
**优化：** 在每个 `workflow_id` 维度加一个**读穿缓存 / 上下文仓库**（agent 写入、按需读取）。既降延迟又省钱，还顺手解决了"各 agent 重复查库"的问题。

### P2-3｜智能体产出缺外部质量门禁（QA 闭环）
**问题：** `SuggestionAgent` 用 `validate_action` 工具做**自校验**（好），但没有任何**外部**质量评估——比如拿"检测出的问题"和 demo Excel 里埋的 11 个已知问题比精度/召回。
**优化：** 诊断后插一个轻量**评估节点**（已知问题集做 golden set），并把"低置信度建议"标记为需人工 triage 而非直接呈现。这正是一个质量门禁（和我的 EvidenceQA 思路一致），能显著提升演示可信度。

### P3-1｜最贵的一步之前没有人工"范围确认"
**问题：** 最烧 token 的 `content_diagnosis` 在人工介入前就跑了。若 `diagnosis_planning` 范围定错了，白跑 50 次诊断。
**优化：** 在 `diagnosis_planning_node` 后**可选**加一个 `interrupt()`，让人确认/微调诊断范围再花 token。复用现有 interrupt 模式，成本极低。

### P3-2｜State 在变胖、工具散落
**问题：** `state.completed_steps` / `state.executed_nodes`（list[dict]）会随规模膨胀 checkpoint；tool 函数是按 agent 散落在 `tools/` 里的。
**优化：** State 只放**协调信号**，体积大的 payload 落到 `workflow_event`/独立表；工具收敛到一个带元数据的**工具注册表**（归属 agent、是否有副作用、成本），方便后续编排与管控。

## 三、优化后的协同拓扑（见附图）

核心变化三处（图中虚线/高亮）：
1. `content_diagnosis` 由串行单步 → **并行 fan-out（map-reduce，受控并发）**，且逐工具调用发事件；
2. `validate_action` 失败 → **回流到 `generate_suggestion`** 带反馈重生成（封顶重试）；
3. 智能体层共享**工具缓存 + 逐调用事件流**，规划可滚动自适应。

> 注意：审核后路由、校验后路由这些**确定性分支保持规则路由**，不要改成 LLM 决策——那是评审报告里已经确认的正确取舍。

## 四、建议落地顺序

| 次序 | 动作 | 收益 | 风险 |
|------|------|------|------|
| 1 | P0-1 给 agent loop 加逐工具调用事件 + 候选级续跑 | 可观测 + 不丢成果 | 低 |
| 2 | P0-2 候选诊断并行 fan-out | 速度数量级提升 | 中（需管并发/限流） |
| 3 | P1-1 校验失败回流重生成 | 真正的协同闭环 | 低 |
| 4 | P2-3 诊断后质量门禁（golden set） | 演示可信度 | 低 |
| 5 | P1-2 / P3-1 滚动规划 + 范围确认 HITL | 省钱 + 可控 | 中 |
| 6 | P2-1 / P2-2 / P3-2 模型路由 / 共享缓存 / State 瘦身 | 韧性 + 成本 | 中 |

---

## 五、一句话总结

**你们已经是一个"编排得当的多代理系统"了——主控是 LangGraph 固定脊，四个智能体节点在关键处做 LLM 决策与 tool calling。下一步不是换架构，而是把"智能体节点内部"做得更可观测、可并行、可自我修正、可质量门禁。** 这正好也是我作为编排者最看重的：每个代理产出都要过质量关，链路失败要能回流重试，而不是一路裸奔到终点。
