> 功能编号：F05
> **里程碑归属：M3（建议生成智能体 + 人工审核闭环）**
> 独立测试目标：根据结构诊断和内容诊断问题，通过 LLM Agent Loop 生成可校验、可审核、可执行的结构化维护建议。

---

## 1. 功能目标

将 `diagnosis_issue` 转换为 `adjustment_suggestion`。每条建议必须包含问题来源、动作类型、目标节点、原因、建议说明、风险等级、置信度和是否需要人工确认。建议只描述动作，不直接修改节点、数据库或 Excel。

---

## 2. 功能边界

### 2.1 输入

1. `version_id`
2. `diagnosis_issue` 待处理问题
3. 节点详情与上下文路径
4. 相似节点结果，可选
5. LLM 配置，可选

### 2.2 输出

1. `adjustment_suggestion` 记录。
2. 可供前端展示的建议列表。

### 2.3 不包含

1. 不执行建议。
2. 不更新节点版本。
3. 不导出 Excel。
4. 不绕过人工审核。

## 5. 动作类型

| 动作 | 来源问题 | 说明 | 默认人工确认 |
|---|---|---|---|
| `add_node` | `missing_parent` | 新增缺失父节点或中间分类 | 是 |
| `move_node` | `bad_parent_child_relation` | 将节点迁移到更合理父节点下 | 是 |
| `rename_node` | `naming_irregular` | 修改节点名称 | 是 |
| `merge_node` | `semantic_duplicate`、`duplicate_name` | 合并重复节点 | 是 |
| `clean_synonym` | `synonym_pollution` | 清理错误同义词 | 是 |
| `split_subtree` | `wide_node` | 为过宽节点提出拆分方案 | 是 |
| `mark_as_valid` | 重复但合理复用 | 标记合理，不修改节点 | 否 |

## 6. 生成策略

### 6.1 规则型建议

适合确定性问题：

| 问题类型 | 建议动作 | 生成规则 |
|---|---|---|
| `missing_parent` | `add_node` | 从 `category_group_name` 中提取缺失父节点名称，建议补齐 |
| `deep_level` | `mark_as_valid` 或 `move_node` | 默认标记需人工判断，不自动移动 |
| `wide_node` | `split_subtree` | 只生成拆分方案，不自动拆分 |
| `duplicate_name` | `mark_as_valid` | 默认先标记待判断，避免误合并 |

### 6.2 LLM 型建议

适合语义问题：

1. 同义词污染：生成 `clean_synonym`。
2. 父子关系异常：生成 `move_node`。
3. 语义重复：生成 `merge_node` 或 `mark_as_valid`。
4. 命名不规范：生成 `rename_node`。

## 9. 校验规则

生成建议后立即做结构校验：

1. `action_type` 必须在允许枚举中。
2. `issue_id` 必须存在。
3. `target_node_id` 必须存在，`add_node` 的缺失父节点场景除外。
4. `risk_level` 必须是 `low`、`medium`、`high`。
5. `confidence` 必须在 0 到 1 之间。
6. `need_confirm=true` 的建议不能自动执行。
7. `clean_synonym` 的待删除同义词必须存在于原 `syn_list`。
8. `merge_node` 必须包含源节点和目标节点。

## 11. 验收标准

1. 每个建议必须包含问题、原因、动作、风险等级和置信度。
2. 建议必须是结构化 JSON，方便程序校验和执行。

4. LLM 返回非法动作时系统拒绝保存并记录错误。
5. 建议生成不修改任何节点数据。

## 12. 建议生成 Agent Loop 设计（新增·2026-07-05）

> 基于架构评审报告，本章节定义建议生成的 ReAct 循环设计。原 §6 描述的"规则型 + LLM 型"建议生成策略保留，但 LLM 型建议生成改为 Agent Loop。

### 12.1 为什么需要 Agent Loop

建议生成的本质是一个**推理 + 查询 + 自校验过程**：

1. LLM 分析问题节点
2. LLM 调用工具查询上下文（查节点路径、查相似节点、查校验规则）
3. LLM 生成建议
4. 程序校验建议合法性
5. 如果校验失败，LLM 重新生成
6. 输出最终建议

这个"分析→查询→生成→校验→重试"的循环，**不应该是一个单次 service 调用**。单次调用无法体现 LLM 的自主推理和自校验能力。

### 12.2 ReAct 循环伪代码

```python
class SuggestionAgent:
    """建议生成智能体，封装 ReAct 循环"""

    def __init__(self, llm, tools, system_prompt):
        self.llm = llm
        self.tools = tools  # [get_node_detail, get_node_path, get_children,
                            #  search_similar_nodes, validate_action, submit_suggestion]
        self.system_prompt = system_prompt
        self.max_retry = 3  # 自校验失败后的最大重试次数

    def run(self, version_id: int, issues: list) -> list[AdjustmentAction]:
        """
        对每条诊断问题执行 ReAct 循环，生成维护建议。

        Args:
            version_id: 当前版本 ID
            issues: 待处理的诊断问题列表

        Returns:
            维护建议列表
        """
        suggestions = []

        for issue in issues:
            # 规则型建议直接生成（不经过 Agent Loop）
            if self.is_rule_based(issue):
                suggestion = self.generate_rule_suggestion(issue)
                suggestions.append(suggestion)
                continue

            # LLM 型建议通过 Agent Loop 生成
            suggestion = self.generate_llm_suggestion(version_id, issue)
            if suggestion:
                suggestions.append(suggestion)

        return suggestions

    def generate_llm_suggestion(self, version_id, issue) -> AdjustmentAction | None:
        """
        对单条 issue 执行 ReAct 循环：
        1. [Thought] 分析 issue 类型和目标节点
        2. [Action] 调用 get_node_detail 获取节点上下文
        3. [Action] 调用 get_node_path 获取路径上下文
        4. [Action] 调用 search_similar_nodes 获取相似节点
        5. [Thought] 生成建议 JSON
        6. [Action] 调用 validate_action 预校验动作合法性
        7. [Observation] 校验通过/失败
        8. [Thought] 如果校验失败，调整建议内容
        9. [Action] 调用 submit_suggestion 提交建议
        """
        messages = [
            SystemMessage(self.system_prompt),
            HumanMessage(f"问题 ID: {issue.id}\n"
                       f"问题类型: {issue.issue_type}\n"
                       f"节点: {issue.node_name}\n"
                       f"描述: {issue.description}\n"
                       f"请生成维护建议。"),
        ]

        for retry in range(self.max_retry):
            for iteration in range(self.max_iter):
                # LLM 思考并决定下一步
                response = self.llm.invoke(messages)

                # 检查是否已生成建议
                if self.is_final_answer(response):
                    suggestion = self.parse_suggestion(response)
                    # 自校验
                    validation = self.validate_action(suggestion)
                    if validation.is_valid:
                        return suggestion
                    else:
                        # 校验失败，提示 LLM 调整
                        messages.append(HumanMessage(
                            f"校验失败：{validation.reason}。请调整建议后重新生成。"
                        ))
                        break  # 跳出内层循环，进入重试
                else:
                    # 提取 tool call
                    tool_call = self.extract_tool_call(response)
                    if tool_call:
                        observation = self.execute_tool(tool_call, version_id)
                        messages.append(AIMessage(response))
                        messages.append(ToolMessage(observation))

        return None  # 超过最大重试次数
```

### 12.3 Tool 列表

| Tool 名称 | 签名 | 功能 | 何时调用 |
|-----------|------|------|---------|
| `get_node_detail` | `(version_id, category_id) → dict` | 查询节点完整详情 | 分析问题时获取上下文 |
| `get_node_path` | `(version_id, category_id) → str` | 查询节点完整路径 | 判断父子关系时 |
| `get_children` | `(version_id, parent_id) → list[dict]` | 查询直接子节点 | 判断节点粒度时 |
| `search_similar_nodes` | `(version_id, node_text, top_k) → list[dict]` | Qdrant 语义召回 | 判断是否重复时 |
| `validate_action` | `(action_json: str) → dict` | 预校验动作合法性 | 生成建议后自校验 |
| `submit_suggestion` | `(suggestion: dict) → str` | 提交一条维护建议 | 自校验通过后提交 |

### 12.4 自校验规则

生成建议后，LLM 必须调用 `validate_action` 工具自校验。校验规则（保留原 §9）：

| 校验项 | 规则 |
|--------|------|
| `action_type` | 必须在允许枚举中（add_node/move_node/rename_node/merge_node/clean_synonym/split_subtree/mark_as_valid） |
| `issue_id` | 必须存在 |
| `target_node_id` | 必须存在（`add_node` 的缺失父节点场景除外） |
| `risk_level` | 必须是 low/medium/high |
| `confidence` | 必须在 0 到 1 之间 |
| `need_confirm` | true 的建议不能自动执行 |
| `clean_synonym` | 待删除同义词必须存在于原 `syn_list` |
| `merge_node` | 必须包含源节点和目标节点 |
| `move_node` | 不能移动到自身子树下 |

校验失败后：
1. `validate_action` 返回 `{valid: false, reason: "..."}`
2. LLM 接收到失败原因后调整建议内容
3. 重新生成并再次校验
4. 超过 `max_retry` 次仍失败则跳过该 issue

### 12.5 Prompt 模板要点

```
你是产品标准分类体系治理专家。请根据输入的问题节点、完整路径、同义词、相似节点和结构诊断结果，生成可执行的维护建议。

你可以使用以下工具：
- get_node_detail: 获取节点详情
- get_node_path: 获取节点完整路径
- get_children: 获取子节点列表
- search_similar_nodes: 搜索语义相似的节点
- validate_action: 预校验建议动作的合法性
- submit_suggestion: 提交维护建议

工作流程：
1. 分析问题类型和目标节点
2. 调用 get_node_detail 和 get_node_path 获取上下文
3. 如果是重复/相似问题，调用 search_similar_nodes 获取相似节点
4. 生成建议 JSON
5. 调用 validate_action 预校验
6. 如果校验失败，根据失败原因调整建议后重新生成
7. 校验通过后调用 submit_suggestion 提交建议

要求：
1. 不要直接删除节点。
2. 对高风险操作必须标记 need_confirm=true。
3. 输出 JSON，不要输出额外解释。
4. action_type 只能从 add_node、move_node、rename_node、merge_node、clean_synonym、split_subtree、mark_as_valid 中选择。
5. 生成后必须调用 validate_action 自校验。
6. 如果校验失败，必须调整后重试。
```

### 12.6 结构化输出 Schema

```python
from pydantic import BaseModel, Field
from typing import Literal, Optional, List

class AdjustmentAction(BaseModel):
    """维护建议结构化输出"""
    issue_id: int = Field(description="关联的诊断问题 ID")
    action_type: Literal[
        "add_node", "move_node", "rename_node",
        "merge_node", "clean_synonym", "split_subtree", "mark_as_valid"
    ] = Field(description="动作类型")
    target_node_id: int = Field(description="目标节点 ID")
    target_node_name: str = Field(description="目标节点名称")
    old_parent_id: Optional[int] = Field(None, description="原父节点 ID")
    new_parent_id: Optional[int] = Field(None, description="新父节点 ID")
    old_name: Optional[str] = Field(None, description="原节点名称")
    new_name: Optional[str] = Field(None, description="新节点名称")
    synonyms_to_remove: List[str] = Field(default_factory=list, description="待删除同义词")
    reason: str = Field(description="生成原因")
    suggestion: str = Field(description="建议说明")
    risk_level: Literal["low", "medium", "high"] = Field(description="风险等级")
    confidence: float = Field(ge=0.0, le=1.0, description="置信度")
    need_confirm: bool = Field(True, description="是否需要人工确认")
```

### 12.7 规则型 vs LLM 型建议

保留原 §6 的分类，但 LLM 型建议通过 Agent Loop 生成：

| 类型 | 适用问题 | 生成方式 |
|------|---------|---------|
| 规则型 | `missing_parent` → `add_node`；`wide_node` → `split_subtree`；`duplicate_name` → `mark_as_valid` | 直接规则生成，不经过 Agent Loop |
| LLM 型 | `synonym_pollution` → `clean_synonym`；`bad_parent_child_relation` → `move_node`；`semantic_duplicate` → `merge_node`/`mark_as_valid`；`naming_irregular` → `rename_node` | 通过 Agent Loop（ReAct + tool calling + 自校验） |

### 12.8 禁止行为

- **禁止**绕过 Agent Loop 直接调 LLM 做单次生成
- **禁止**让 LLM 生成建议后直接写库——必须先经过 `validate_action` tool 自校验
- **禁止**跳过自校验直接提交建议
- **禁止**自校验失败后不重试直接保存——必须重试或跳过
- **禁止**在 node 函数中拼 prompt——prompt 在 `agents/prompts.py` 中管理

### 12.9 异常处理

| 异常 | 处理方式 |
|---|---|
| LLM 调用失败 | 当前 issue 标记失败，继续处理下一条 |
| LLM 输出非 JSON | 重试一次；仍失败则记录 `MODEL_OUTPUT_INVALID` |
| 自校验失败 | LLM 根据失败原因调整后重试（最多 `max_retry` 次） |
| 超过 max_retry | 跳过该 issue，记录 `SUGGESTION_GENERATION_FAILED` |
| Tool 调用超时 | 跳过当前 tool，LLM 重新决策 |
