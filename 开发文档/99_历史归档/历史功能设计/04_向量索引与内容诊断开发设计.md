## 1. 功能目标

将指定版本的节点文本写入 Qdrant，提供相似节点召回能力；对高价值候选问题调用 LLM 做语义判断，生成内容类 `diagnosis_issue`，为后续智能建议生成提供依据。

---

## 2. 功能边界

### 2.1 输入

1. `version_id`
2. `category_node` 节点集合
3. Qdrant 连接配置
4. Embedding 模型配置
5. LLM 模型配置

### 2.2 输出

1. Qdrant Collection：`taxonomy_nodes`
2. 内容诊断问题：
   - `synonym_pollution`
   - `semantic_duplicate`
   - `bad_parent_child_relation`
   - `inconsistent_granularity`
   - `naming_irregular`
3. 相似节点查询结果。

### 2.3 不包含

1. 不执行任何节点修改。
2. 不生成最终维护动作；维护动作属于 F05。
3. Qdrant 或 LLM 不可用时，不影响 F03 结构诊断。

## 6. 内容诊断策略

### 6.1 同义词污染

候选选择：

1. `syn_list` 非空。
2. 同义词数量大于 3 的节点优先。
3. 节点路径语义明确的叶子节点优先。

LLM 输入：

```json
{
  "category_id": 441,
  "category_name": "苹果",
  "path_names": "农、林、牧、渔业类产品 > 农业产品 > 水果及坚果 > 水果（园林水果） > 苹果",
  "synonyms": ["AirPods", "Apple", "Apple Music", "Apple Pencil", "iPhone"]
}
```

LLM 输出要求：

```json
{
  "is_issue": true,
  "issue_type": "synonym_pollution",
  "abnormal_synonyms": ["AirPods", "Apple Music", "Apple Pencil", "iPhone"],
  "reason": "节点位于水果路径下，但同义词包含电子品牌和设备名称。",
  "risk_level": "medium",
  "confidence": 0.92
}
```

### 6.2 疑似重复节点

候选选择：

1. 名称完全相同的节点。
2. Qdrant TopK 相似度高于 `0.85` 的节点。
3. 排除自身和同一路径下的父子节点。

LLM 结论：

| 结论 | 后续建议方向 |
|---|---|
| 语义重复 | `merge_node` |
| 多视角合理复用 | `mark_as_valid` |
| 需要人工判断 | 保留内容问题 |

### 6.3 父子关系异常

候选选择：

1. 子节点名称与父节点路径语义差异明显。
2. Qdrant 可召回更相似的候选父节点。
3. 规则诊断中已出现孤立或层级异常的路径优先。

LLM 输出包含：

1. 当前父子关系是否合理。
2. 更合理候选父节点。
3. 判断理由。
4. 置信度。

## 7. Prompt 约束

内容诊断 Prompt 必须要求：

1. 只判断是否存在内容问题，不输出修改 SQL。
2. 输出 JSON，不输出额外解释。
3. 置信度范围为 0 到 1。
4. 风险等级只能是 `low`、`medium`、`high`。
5. 不确定时输出 `is_issue=false` 或较低置信度。

---

## 8. 异常处理

| 异常 | 处理方式 |
|---|---|
| Qdrant 连接失败 | 返回 `QDRANT_UNAVAILABLE`，允许继续结构诊断 |
| Embedding 调用失败 | 标记索引任务失败，保留节点数据 |
| LLM 调用失败 | 当前候选标记失败，继续处理下一条 |
| LLM 输出非 JSON | 重试一次；仍失败则记录 `MODEL_OUTPUT_INVALID` |
| 相似节点为空 | 返回空数组，不视为错误 |

## 11. 内容诊断 Agent Loop 设计（新增·2026-07-05）

> 基于架构评审报告，本章节定义内容诊断的 ReAct 循环设计。原 §6 描述的"召回→LLM判断"流程保留作为候选选择策略，但诊断执行改为 Agent Loop。

### 11.1 为什么需要 Agent Loop

内容诊断的本质是一个**迭代探索过程**：

1. 对每个候选节点，召回相似节点
2. LLM 判断是否有问题
3. 如果 LLM 不确定，需要补充查询（如查父节点路径、查兄弟节点）
4. LLM 再次判断
5. 输出结论

这个"召回→判断→补充查询→再判断"的循环，**不应该在一个单次 service 调用中完成**，而应该是一个 agent loop。单次 service 调用无法体现 LLM 的自主决策能力。

### 11.2 ReAct 循环伪代码

```python
class ContentDiagnosisAgent:
    """内容诊断智能体，封装 ReAct 循环"""

    def __init__(self, llm, tools, system_prompt):
        self.llm = llm
        self.tools = tools  # [search_similar_nodes, get_node_detail, get_node_path,
                            #  get_children, submit_diagnosis]
        self.system_prompt = system_prompt

    def run(self, version_id: int, plan: DiagnosisPlan) -> list[ContentIssue]:
        """
        对每个候选节点执行 ReAct 循环。

        Args:
            version_id: 当前版本 ID
            plan: 诊断规划结果（来自 diagnosis_planning_node）

        Returns:
            内容诊断问题列表
        """
        candidates = self.select_candidates(version_id, plan)
        issues = []

        for candidate in candidates:
            issue = self.diagnose_single_node(version_id, candidate)
            if issue:
                issues.append(issue)

        return issues

    def diagnose_single_node(self, version_id, candidate) -> ContentIssue | None:
        """
        对单个候选节点执行 ReAct 循环：
        1. [Thought] 分析候选节点
        2. [Action] 调用 search_similar_nodes 查相似节点
        3. [Observation] 获得相似节点列表
        4. [Thought] 判断是否需要查父节点路径
        5. [Action] 调用 get_node_path 查路径
        6. [Observation] 获得完整路径
        7. [Thought] 综合判断是否存在语义问题
        8. [Action] 调用 submit_diagnosis 提交结论
        """
        messages = [
            SystemMessage(self.system_prompt),
            HumanMessage(f"请诊断节点：{candidate.category_name}，路径：{candidate.path_names}"),
        ]

        for iteration in range(self.max_iter):
            # LLM 思考并决定下一步
            response = self.llm.invoke(messages)

            # 检查是否已得出结论
            if self.is_final_answer(response):
                return self.parse_output(response)

            # 提取 tool call
            tool_call = self.extract_tool_call(response)
            if tool_call:
                observation = self.execute_tool(tool_call, version_id)
                messages.append(AIMessage(response))
                messages.append(ToolMessage(observation))
            else:
                # 无 tool call 也无最终答案，提示 LLM 继续
                messages.append(HumanMessage("请继续分析或提交诊断结论。"))

        return None  # 超过最大迭代次数
```

### 11.3 Tool 列表

| Tool 名称 | 签名 | 功能 | 何时调用 |
|-----------|------|------|---------|
| `search_similar_nodes` | `(version_id, node_text, top_k=10) → list[dict]` | Qdrant 语义召回相似节点 | 初次分析时召回 |
| `get_node_detail` | `(version_id, category_id) → dict` | 查询节点完整详情 | 需要节点上下文时 |
| `get_node_path` | `(version_id, category_id) → str` | 查询节点完整路径 | 判断父子关系时 |
| `get_children` | `(version_id, parent_id) → list[dict]` | 查询直接子节点 | 判断节点粒度时 |
| `submit_diagnosis` | `(issue: dict) → str` | 提交一条诊断结果 | 得出结论时 |

### 11.4 Prompt 模板要点

```
你是产品标准分类体系的内容诊断专家。

你的任务是对指定的分类节点进行内容诊断，识别以下类型的问题：
1. synonym_pollution: 同义词污染（如水果节点混入电子品牌词）
2. semantic_duplicate: 语义重复（不同路径下的相同/相似节点）
3. bad_parent_child_relation: 父子关系异常（子节点与父节点语义不匹配）
4. inconsistent_granularity: 分类粒度不一致
5. naming_irregular: 命名不规范

你可以使用以下工具：
- search_similar_nodes: 搜索语义相似的节点
- get_node_detail: 获取节点详情
- get_node_path: 获取节点完整路径
- get_children: 获取子节点列表
- submit_diagnosis: 提交诊断结论

工作流程：
1. 先调用 search_similar_nodes 查找相似节点
2. 根据需要调用 get_node_path 或 get_children 补充上下文
3. 综合判断是否存在内容问题
4. 如果存在问题，调用 submit_diagnosis 提交结论
5. 如果没有问题，明确说明原因

输出要求：
- 只判断是否存在内容问题，不输出修改 SQL
- 置信度范围为 0 到 1
- 风险等级只能是 low、medium、high
- 不确定时输出 is_issue=false 或较低置信度

示例诊断结果 JSON：
{
  "is_issue": true,
  "issue_type": "synonym_pollution",
  "abnormal_synonyms": ["AirPods", "Apple Music", "Apple Pencil", "iPhone"],
  "reason": "节点位于水果路径下，但同义词包含电子品牌和设备名称。",
  "risk_level": "medium",
  "confidence": 0.92
}
```

### 11.5 结构化输出 Schema

```python
from pydantic import BaseModel, Field
from typing import Literal

class ContentDiagnosisOutput(BaseModel):
    """内容诊断结构化输出"""
    is_issue: bool = Field(description="是否存在内容问题")
    issue_type: Literal[
        "synonym_pollution",
        "semantic_duplicate",
        "bad_parent_child_relation",
        "inconsistent_granularity",
        "naming_irregular"
    ] | None = Field(None, description="问题类型")
    abnormal_synonyms: list[str] = Field(default_factory=list, description="异常同义词列表")
    reason: str = Field("", description="判断理由")
    risk_level: Literal["low", "medium", "high"] = Field("low", description="风险等级")
    confidence: float = Field(0.0, ge=0.0, le=1.0, description="置信度")
```

### 11.6 候选选择策略

候选节点来源（保留原 §6 策略，但由 `diagnosis_planning_node` 的 plan 指导）：

1. **同义词污染候选**：`syn_list` 非空且同义词数量 > 3 的节点优先
2. **疑似重复候选**：名称完全相同的节点 + Qdrant TopK 相似度 > 0.85 的节点
3. **父子关系异常候选**：规则诊断中已出现孤立或层级异常的路径优先

`diagnosis_planning_node` 输出的 `DiagnosisPlan` 指导：
- `priority_subtrees`：优先诊断哪些子树
- `sample_strategy`：`focused`（聚焦高优先级）/ `full_scan`（全量）/ `sampling`（采样）
- `estimated_candidates`：预估候选数量

### 11.7 禁止行为

- **禁止**绕过 Agent Loop 直接调 LLM 做单次判断
- **禁止**让 LLM 直接写 `diagnosis_issue` 表——通过 `submit_diagnosis` tool 提交，由 service 写库
- **禁止**全量扫描 21090 个节点——必须先经 `diagnosis_planning_node` 规划
- **禁止**在 node 函数中拼 prompt——prompt 在 `agents/prompts.py` 中管理
- **禁止**在 node 函数中直接操作 Qdrant——通过 service + tool

### 11.8 异常处理

| 异常 | 处理方式 |
|---|---|
| LLM 调用失败 | 当前候选标记失败，继续处理下一条 |
| LLM 输出非 JSON | 重试一次；仍失败则记录 `MODEL_OUTPUT_INVALID` |
| Tool 调用超时 | 跳过当前 tool，LLM 重新决策 |
| ReAct 循环超过 max_iter | 返回 `None`，不生成诊断结果 |
| Qdrant 连接失败 | 返回 `QDRANT_UNAVAILABLE`，允许继续结构诊断 |
