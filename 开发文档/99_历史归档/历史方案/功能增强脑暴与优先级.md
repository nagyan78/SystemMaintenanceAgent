# 功能增强脑暴与优先级（打破"流水线单调感"）

> 背景：当前系统是「导入 → 诊断 → 单次审核 → 出版本 → 报告 → 结束」的单向、单人、单次流水线。
> 要打破单调，需补三类"非流水线"能力：**主动闭环 / 多人协作 / AI 能力产品化**，并让结果**可读可用**。
> 所有新增功能必须服务于"维护产品分类树"这一核心，不要做成聊天玩具。

## 为什么现在单调（根因）

1. **无回头**：graph 仅 `wait_human_review_node` 一个 interrupt，跑完即终态，无法二次审核/续跑。
2. **无循环**：版本是静态快照，无定时/事件触发复检（详见《审核多轮与版本续跑修复方案》）。
3. **单人单角色**：审核只有"通过/驳回"，无多角色、无评论、无协作。
4. **AI 不可见**：Qdrant 索引、agent loop 只服务于"诊断+建议"内部，用户感知不到 AI 的存在。
5. **结果孤立**：版本链、诊断问题表已存在，但缺少 diff 可视化与健康洞察。

---

## 方向 A — 让系统"主动"起来（自动化 / 闭环）

| 编号 | 功能 | 解决的问题 | 复用资产 | 工作量 | 优先级 |
|------|------|-----------|---------|--------|--------|
| A1 | **定时 / 事件触发巡检** | 几天后想复查不必手动重跑；文件变更自动增量诊断 | deterministic spine + from_version 续跑 + SSE | M | **高**（接之前架构方案） |
| A2 | **改动影响分析（What-if 沙箱）** | 执行前不知道"改这个会牵连多少下游"，不敢点批准 | version_service + 执行逻辑复用 | M | 中高 |
| A3 | **语义去重 / 相似类目巡检** | 规则只查精确重复，漏掉"同义不同名"类目 | Qdrant 向量召回 | S–M | 中 |

## 方向 B — 让"人"更好地协作（多角色 / 审计）

| 编号 | 功能 | 解决的问题 | 复用资产 | 工作量 | 优先级 |
|------|------|-----------|---------|--------|--------|
| B1 | **多角色审核流 + 评论线程** | 业务专家与架构师关注点不同，单次二值审核不够 | wait_human_review interrupt + suggestion 状态机 | M–L | 中（团队用才值） |
| B2 | **操作审计时间线** | 谁、何时、改了什么、为何，事后说不清 | operation_log 表已存在 | S | 中 |
| B3 | **多租户 / 多产品线** | 不同产品线各自分类树混在一起 | file / task 模型 | L | 低（除非有需求） |

## 方向 C — 把已有 AI 能力产品化（Qdrant / Agent 复用）

| 编号 | 功能 | 解决的问题 | 复用资产 | 工作量 | 优先级 |
|------|------|-----------|---------|--------|--------|
| C1 | **分类咨询 Copilot** | "这个商品该归哪类？"——让 Qdrant+LLM 直接面向用户产生交互价值 | Qdrant + chat API 占位(当前 501) | M | **高**（让 AI 可见） |
| C2 | **规则库自学习** | 固定规则漏判，用户想把"这类总该标"沉淀为自定义规则 | structure_diagnosis 规则层 | M–L | 中 |
| C3 | **智能补全 / 建议草稿** | 新增类目时人工想父类、同义词费劲 | generate_suggestion agent | S–M | 中 |

## 方向 D — 把"结果"变得可读可用（洞察 / 导出）

| 编号 | 功能 | 解决的问题 | 复用资产 | 工作量 | 优先级 |
|------|------|-----------|---------|--------|--------|
| D1 | **版本对比可视化 Diff 树** | v1.0 → v1.1 增删改移动看不清 | version chain | M | **高**（版本迭代配套） |
| D2 | **分类健康看板** | issue 密度、孤儿节点、覆盖率及随版本趋势无视图 | diagnosis_issue 表 | M | 中高 |
| D3 | **面向业务的变更说明书** | 技术报告业务方看不懂 | generate_report | S | 中 |

---

## 推荐先做的前 3 个（价值高、复用强、连到已有规划）

1. **A1 定时/事件巡检** —— 就是之前"自动复检闭环"的落地，补全版本续跑后顺理成章。
2. **D1 版本 Diff 可视化** —— 版本迭代的必需配套，否则多版本只是数据库里的数字。
3. **C1 分类咨询 Copilot** —— 当前 chat API 是 501 占位，这是让 Qdrant/LLM 真正产生用户可感知价值的最佳落点，也最"不单调"。

> 落地顺序建议：先完成《审核多轮与版本续跑修复方案》的 PR1/PR2（止血+循环），再上 A1/D1；C1 可与 PR3 并行。

## 一条明确的"别动"提醒

- 不要把"审核后/校验后"的确定性分支改成 LLM 决策（架构评审已定）。
- 不要为了热闹加"通用聊天机器人"——所有功能必须服务分类树维护。
- 自动复检默认**不**全量重跑，用 `from_version` 增量诊断 + deferred 候选集，避免重复烧 token。
