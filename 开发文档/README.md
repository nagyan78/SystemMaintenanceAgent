# 开发文档总入口

> 状态：CURRENT
> 更新日期：2026-07-17
> 本文件是项目开发文档的唯一入口，负责定义阅读顺序、文档优先级和历史资料使用规则。

## 1. 文档优先级

发生描述冲突时，按以下顺序裁决：

1. 当前代码与自动化测试；
2. [当前实现情况](00_当前状态/当前实现情况.md)；
3. [当前开发路线图](00_当前状态/当前开发路线图.md)；
4. 与任务直接相关的当前 PRD 和 R1～R3 执行计划；
5. 产品需求与目标架构参考；
6. `99_历史归档/` 中的历史资料。

未提交的工作树改动可以作为“本地当前事实”，但不等于能力已经发布。

## 2. 最小阅读顺序

修改代码前至少阅读：

1. 本文件；
2. [当前实现情况](00_当前状态/当前实现情况.md)；
3. [当前开发路线图](00_当前状态/当前开发路线图.md)；
4. 与任务直接相关的功能需求或执行计划；
5. 涉及长期架构变化时，再阅读[多智能体目标架构](01_产品与架构/多智能体目标架构.md)。

[R1 可信诊断与完整结果](03_开发执行计划/R1_可信诊断与完整结果_执行计划.md)和阶段五 R1～R3 主闭环已经验收完成；当前优先级以[当前开发路线图](00_当前状态/当前开发路线图.md)中的 R4 可靠性与规模化增强为准。

## 5. 历史资料使用规则

- 历史文档中的接口、schema、算法和决策背景可以参考，但不能直接代表当前实现。
- 历史文档出现“当前”“最终”“必须先做”等措辞时，不自动获得更高优先级。
- 使用历史方案前，必须与代码、测试和 `00_当前状态/` 下的当前文档核对。
- 如果历史设计仍有价值，应把结论提炼到当前 PRD 或执行计划中，而不是要求开发者拼接多份旧文档。

详见[历史归档说明](99_历史归档/README.md)。
